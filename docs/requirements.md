## 概要

チャットの問い合わせ内容に応じて、応答を表・グラフなどのUI部品として動的に生成・表示する機能を、既存のChatGPT風試作アプリに追加する。応答は「テキスト＋UIコンポーネント配列」の構造化JSONスペックとしてバックエンドから返し、フロントエンドが事前に用意した描画コンポーネント（表・棒グラフ）でレンダリングする。本フェーズではLLMには接続せず、バックエンドのキーワードマッチによるモック実装（現行 `EchoService` の置き換え）が構造化応答を返す。JSONスキーマは将来LLMの出力契約としてそのまま使える形で設計・文書化する。

背景: 「今月の問い合わせ件数を担当者別にまとめて」のような自然文の問いに対し、結果が表やグラフとして動的に生成される体験を検証したい。将来的にはLLMがこの構造化応答を生成する構想であり、本フェーズはその前段として、(1) 応答スキーマ（＝将来のLLM出力契約）の確立、(2) フロントエンドの動的描画基盤の実装、をモックで先行させる。

## スコープ（機能要件）

- FR-1: `POST /api/chat` は、応答テキストと0個以上のUIコンポーネントスペックの配列からなる構造化JSONを返す。従来の `{"reply": string}` 形式は廃止する
- FR-2: UIコンポーネントスペックのtypeとして「表（table: 列名配列＋行データ）」と「棒グラフ（bar_chart: ラベル配列＋数値配列＋タイトル等）」の2種をサポートする
- FR-3: 1つの応答に複数のUIコンポーネント（例: 表＋棒グラフ）を併記できる
- FR-4: バックエンドのモックサービスは入力文のキーワードマッチで以下の2シナリオを出し分ける（データは架空の「問い合わせ管理」データで固定）
  - FR-4a: 「担当者」系キーワード → 担当者別問い合わせ件数（要約テキスト＋表＋棒グラフ）
  - FR-4b: 「カテゴリ」「種別」系キーワード → カテゴリ別問い合わせ件数（要約テキスト＋表＋棒グラフ）
- FR-5: どのキーワードにもマッチしない入力には、UIコンポーネントなしの固定の使い方案内テキスト（例:「『今月の問い合わせ件数を担当者別にまとめて』のように聞いてください」）を返す
- FR-6: フロントエンドはアシスタント応答の吹き出し内に、応答テキストに続けて各UIコンポーネントを順に描画する（table → HTMLテーブル、bar_chart → 棒グラフ）
- FR-7: フロントエンドが解釈できない応答（未知のtype、必須フィールド欠落等）を受け取った場合、「応答を表示できませんでした」等のエラー文言を吹き出しとして表示する（アプリはクラッシュしない）
- FR-8: 通信エラー時は既存どおりエラーメッセージの吹き出しを表示する（現行動作を維持）
- FR-9: UIコンポーネントを含むメッセージは会話履歴（メモリ内）に保持され、サイドバーで会話を切り替えて戻った際に表・グラフも再表示される
- FR-10: 応答JSONスキーマを、将来LLMにそのまま出力させる契約として `docs/` 配下に文書化する（フィールド定義・制約・サンプルを含む）

### 影響範囲

- フロントエンド型定義 — `frontend/src/types/chat.ts`: `Message` に構造化応答（UIコンポーネントスペック）を保持するフィールドを追加
- フロントエンドAPIクライアント — `frontend/src/api/chatApi.ts`: 新レスポンス形式への対応
- フロントエンド状態管理 — `frontend/src/stores/conversationStore.ts`: 構造化応答をメッセージとして格納する処理
- フロントエンド画面コンポーネント — `frontend/src/components/MessageBubble.vue` の拡張＋表・棒グラフ描画コンポーネントの新規作成。グラフ描画ライブラリの新規依存追加（選定は設計フェーズに委ねる）
- バックエンドAPI — `ChatController` / `ChatResponse`（レスポンスDTOの再設計）
- バックエンドサービスロジック — `EchoService` を置き換えるモック応答サービス＋そのテスト
- ドキュメント — 応答JSONスキーマの契約文書（`docs/` 配下、新規）

DBスキーマへの影響はなし（バックエンドは引き続きステートレス）。

### 画面・UIフロー

既存のチャット画面（Sidebar ＋ ChatWindow ＋ MessageInput）に新規画面は追加しない。変わるのはアシスタント側の吹き出し（MessageBubble）の中身のみ:

1. ユーザーがメッセージを入力・送信（既存フロー通り。送信中ローディングも既存通り）
2. 応答受信後、アシスタント吹き出しに要約テキスト、その下に表・棒グラフが縦に並んで表示される
3. 非マッチ時・エラー時は従来同様テキストのみの吹き出し
4. 会話切り替え・New Chat の挙動は既存のまま（リッチUIも履歴として復元される）

## 対象外（Non-goals）

- 実際のLLM API連携（本フェーズはモックのみ。スキーマは連携を見据えて設計するが接続はしない）
- 折れ線グラフ・円グラフなど、表・棒グラフ以外のUI部品
- 表・グラフのインタラクティブ機能（ソート、フィルタ、CSVエクスポート、ホバー詳細等）— 静的な表示のみ
- 実データソース（DB・外部API）への接続。データはモック内の架空データのみ
- ストリーミング応答、認証、会話の永続化（既存プロジェクト方針を踏襲）
- 従来のエコー応答の維持（本機能で廃止する）

## 受け入れ基準

- AC-1: 「今月の問い合わせ件数を担当者別にまとめて」と送信すると、要約テキスト・担当者別件数の表・棒グラフが1つの応答吹き出し内に表示される（FR-2, 3, 4a, 6。検証方法: 手動確認）
- AC-2: 「カテゴリ別の問い合わせ件数を見せて」と送信すると、AC-1とは異なるデータ（カテゴリ別）の表＋棒グラフが表示される（FR-4b。検証方法: 手動確認）
- AC-3: 「こんにちは」等キーワード非マッチの入力には、表・グラフなしの使い方案内テキストのみが表示され、「エコー: 」応答は返らない（FR-5。検証方法: 自動テスト＋手動確認）
- AC-4: `POST /api/chat` に対する統合テストで、担当者系キーワード入力時に table と bar_chart を含む構造化JSON、非マッチ入力時にコンポーネント0件のJSONが返ることを確認できる（FR-1, 2, 4。検証方法: 自動テスト（MockMvc））
- AC-5: 未知のtypeを含む応答スペックをフロントに与えた場合、エラー文言の吹き出しが表示され、画面全体は操作可能なまま維持される（FR-7。検証方法: 手動確認（モックデータ差し替え等の一時的な手段でよい））
- AC-6: バックエンド停止状態で送信すると、既存どおりエラーメッセージの吹き出しが表示される（FR-8。検証方法: 手動確認）
- AC-7: 表・グラフを含む会話から別会話へ切り替えて戻ると、表・グラフが再描画される（FR-9。検証方法: 手動確認）
- AC-8: 応答JSONスキーマの契約文書が `docs/` 配下に存在し、AC-4のテストが検証する実際のレスポンスと一致している（FR-10。検証方法: 手動確認）
- AC-9: 空文字・空白のみの送信は従来どおり拒否される（バックエンド400／フロント送信抑止の現行動作を退行させない。検証方法: 自動テスト。現状 `ChatRequest.@NotBlank` によるバックエンド400と `MessageInput.canSend` によるフロント送信抑止で実現済み — 退行させないこと）

## 制約

- 技術スタックは既存踏襲で変更不可: フロント Vue 3 + TypeScript + Pinia（Vite）、バックエンド Java Spring Boot + Maven
- バックエンドはステートレスを維持する（会話状態・セッションを持たない。既存設計方針）
- フロント→バックの通信は既存の Vite dev プロキシ（`/api` → `localhost:8080`）経由。CORS設定は追加しない
- 表・棒グラフ描画のための新規フロント依存の追加は許容する（ライブラリ選定は設計フェーズに委ねる。軽量であること）。現状 `frontend/package.json` にグラフ描画ライブラリの依存はない
- 過度な作り込みをせず「動くプロトタイプ」を優先する既存方針を踏襲する

### 重大な非機能要件

- **表示の安全性（XSS防止）**: 応答スペック内の文字列（テキスト、列名、セル値、ラベル等）は、将来LLM由来の値になる前提のため、HTMLとして解釈せず必ずテキスト／描画データとして扱うこと。`v-html` 等による生HTML描画は用いない
- **不正入力への耐性**: 将来LLMが崩れたJSON・想定外の構造を返すケースの布石として、フロントは不正な応答スペックでクラッシュせずエラー表示にフォールバックすること（FR-7）

### 前提（調査により確認）

- 利用者は開発者本人およびデモ閲覧者のみ。認証・ロールによる挙動分岐はなし（既存プロジェクトのスコープ外方針を踏襲）
- `POST /api/chat` のレスポンス形式変更は破壊的変更でよい（APIの利用者は本フロントエンドのみで、バージョニングや互換維持は不要）
- 既存データの移行は不要（永続化されたデータが存在しないため）
- 会話・応答はメモリ内（Piniaストア）のみで保持し、リロードで消えることを許容する（既存方針）
- モックの架空データ（担当者名・カテゴリ名・件数）の具体値は設計・実装フェーズで自由に決めてよい
- 送信中に別会話へ切り替えた場合の挙動は既存設計（応答は送信元会話にのみ反映）を踏襲する

### 決定記録（正本メモより）

- Q1: 動的UIの記述方式 → 構造化JSONスペック（安全でテスト可能、将来LLMにJSONを出力させやすく、既存のコンポーネント構成とも親和）
- Q2: モック応答の実装場所 → バックエンド（既存の`EchoService`置き換えで済み、将来ここをLLM呼び出しに差し替えるだけになる）
- Q3: 対応UI部品 → 表＋棒グラフ（メモの例「担当者別件数を表やグラフで」を満たす最小セット）
- Q4: モックの応答決定方法 → キーワードマッチ＋数シナリオ（「問いに応じてUIが変わる」体験を最小コストで再現できる）
- Q5: キーワード非マッチ時の挙動 → テキストのみで応答（本物のLLMも雑談にはテキストで返すため自然）
- Q6: 複数UI部品の併記 → 許可（メモの「表やグラフ」に直接応え、スキーマを配列にするだけでコスト増が小さい）
- Q7: 解釈不能な応答の表示 → エラー文言を吹き出し表示（既存の通信エラー時と同じパターンで揃う）
- Q8: 表・グラフのインタラクティブ機能 → 含めない・表示のみ（今回の検証目的は動的UI生成の骨格であり対話機能は後回しでよい）
- Q9: 非マッチ時のテキスト内容 → 使い方の案内文とし、エコー応答は廃止（デモを見た人が次に何を打てばよいか分かる）
- Q10: デモシナリオ構成 → 2シナリオ（担当者別／カテゴリ別。キーワードによる出し分けの仕組み自体を最小コストでデモできる）
- Q11: 応答JSONスキーマの位置づけ → 将来のLLM出力契約として設計・文書化（「最初はモックで」というメモの意図＝いずれ本物に繋ぐ、に沿う）

## 用語集

- **問い合わせ**: 本書では2つの意味で登場する。(1)「LLMへの問い合わせ」＝ユーザーがチャットに入力する自然文の質問、(2)「問い合わせ件数」＝デモ用架空データの題材である、顧客からの問い合わせ管理業務の件数。FR-4等の「問い合わせ管理データ」は後者を指す
- **動的UI生成**: LLM（本フェーズではモック）が任意のコードを生成・実行することではなく、宣言的なJSONスペックを返し、フロントエンドの既製コンポーネントがそれを描画すること
- **モック**: バックエンド内のキーワードマッチによる決定論的な応答生成ロジック。将来LLM API呼び出しに置き換えられる部分
- **UIコンポーネントスペック**: 応答JSON内の1つの描画指示（type＋データ）。表・棒グラフのいずれか

## 未決定事項（Open Questions）

- 実LLM連携時のデータの出所 — 選択肢: A) LLMが応答スペックと集計データを一体で生成 / B) LLMはUIスペックのみ生成し、データはバックエンドが実データソース（DB等）から供給。推奨仮置き: A（本フェーズのスキーマはデータ内包型であり、モックからの差し替えが最小になるため）。外れた場合の影響: Bになる場合、スキーマにデータ参照・クエリ結果の埋め込み方式の拡張が必要となり、契約文書とフロント描画層の一部改修が発生する。確認先: 実LLM連携タスクの起案時にプロダクトオーナー（利用するLLMプロバイダ・実データソースの有無とあわせて決定）。**本フェーズ（モック実装・スキーマ設計）の遂行には影響しないため、設計フェーズをブロックしない**

上記以外に、コード突合の結果として設計フェーズの意思決定に影響する新規の曖昧点は見つからなかった。
