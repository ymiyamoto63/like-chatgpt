# 新規問い合わせ機能（チャット対話式・選択肢ウィザード）

## 概要

サイドバーに「新規問い合わせ」ボタンを追加し、押下すると新規会話上で問い合わせチケットの新規登録ウィザードが始まる。アシスタント（バックエンドMock）が質問と選択肢を返し、ユーザーは原則ボタン選択で回答を進める（カテゴリ→緊急度→内容入力→確認→受付完了）。選択肢はチャット応答スキーマに新設する「選択肢コンポーネント」としてバックエンドから返し、フロントエンドが吹き出し内に選択ボタンとして描画する。実際のLLMには接続せず、既存方針どおりバックエンドのステートレスなMockが対話フローを駆動する。

## 背景・目的

チャットUI上で「対話的に情報を集めて処理を完了する」体験（フォームの代替としてのチャットウィザード）をモックで検証する。将来LLMに差し替えた際に「LLMが選択肢つきの質問を返し、ユーザーの回答を受けて対話を進める」構図をそのまま実現できるよう、対話の主導権をバックエンド（＝将来のLLM）側に置き、選択肢を応答スキーマの一部として確立することが狙い。入力はなるべく選択式にし、自由入力を問い合わせ内容のみに限定する。

## スコープ

- フロントエンド: [Sidebar.vue](../../frontend/src/components/Sidebar.vue) 上部（New Chat ボタン直下）に「新規問い合わせ」ボタンを追加
- フロントエンド: 選択肢コンポーネントの描画（吹き出し内の選択ボタン群）と、クリック時の定型文送信
- フロントエンド: 最新アシスタント発言に選択肢がある間のテキスト入力欄の無効化、過去の選択肢ボタンの無効化
- フロントエンド: 内容入力ステップで全回答を累積した定型形式で送信する処理
- フロントエンド: [chatApi.ts](../../frontend/src/api/chatApi.ts) の応答検証と [types/chat.ts](../../frontend/src/types/chat.ts) への選択肢コンポーネント型の追加
- バックエンド: [MockChatService.java](../../backend/src/main/java/com/example/chatbackend/MockChatService.java) にウィザードフロー（開始／カテゴリ→緊急度→内容→確認→完了／キャンセル・やり直し）の応答分岐を追加
- バックエンド: 選択肢コンポーネントの `UiComponent` 実装追加（`@JsonSubTypes` 拡張）とテスト
- ドキュメント: [chat-response-schema.md](../chat-response-schema.md) に選択肢コンポーネントの契約を追記

## 対象外（Non-goals）

- 実際のLLM API連携（本フェーズはMock。スキーマは連携を見据えて設計するが接続はしない）
- 登録された問い合わせデータの保存・一覧・検索（受付完了の表示のみで、どこにも永続化・蓄積しない）
- 確認ステップでの項目単位の修正（「やり直す」は最初からの再入力のみ）
- 添付ファイル・画像などテキスト以外の入力
- 既存の定型レポートボタン・キーワードマッチ応答（担当者別／カテゴリ別）の仕様変更
- 認証・ロール分岐、会話の永続化（既存方針踏襲）

## 機能要件

- FR-1: サイドバー上部の New Chat ボタン直下に「新規問い合わせ」ボタンを表示する。応答待ち中（`isLoading`）は無効化する
- FR-2: ボタン押下で常に新規会話を作成・アクティブ化し、開始トリガー文（固定の定型文。例「新規問い合わせ」）をユーザー発言として `POST /api/chat` に送信する。Mockは最初の質問（カテゴリ選択）を選択肢つき応答で返す
- FR-3: チャット応答スキーマに選択肢コンポーネント（type名は設計フェーズで決定。例 `choices`）を追加する。質問の提示テキストは `reply`、選択肢（表示ラベル＝クリック時の送信文、1件以上）はコンポーネントが持つ。契約文書に追記する
- FR-4: フロントエンドは選択肢コンポーネントをアシスタント吹き出し内の選択ボタン群として描画する。ボタンクリックでそのラベルに対応する定型文（例「カテゴリ: 請求」）がユーザー発言として送信され、Mockが次のステップの応答を返す
- FR-5: ウィザードは次のステップで構成する:
  1. カテゴリ選択（請求／技術／アカウント／その他 ＋ キャンセル）
  2. 緊急度選択（高／中／低 ＋ キャンセル）
  3. 内容の自由入力（選択肢なし応答。テキスト入力欄が有効になる）
  4. 確認（入力内容の要約を表コンポーネントで提示 ＋「登録する」「やり直す」「キャンセル」）
  5. 受付完了（架空の受付番号を含む完了メッセージ、選択肢なし）
- FR-6: 内容入力ステップでは、フロントエンドがそれまでの回答を累積した定型形式（例「カテゴリ: 請求 / 緊急度: 高 / 内容: <入力文>」）に組み立てて送信し、Mockはこれを解析して確認ステップ（要約表＋選択肢）を返す。組み立て後の文字列がユーザー吹き出しにそのまま表示される
- FR-7: 各選択ステップの「キャンセル」押下でMockが中断メッセージ（選択肢なし）を返し、フローは終了する。以降その会話は通常のチャットとして継続入力できる
- FR-8: 確認ステップで「登録する」を選ぶと受付完了メッセージ（架空の受付番号つき）が返る。「やり直す」を選ぶと同じ会話内でカテゴリ選択から再開する。登録データはどこにも保存しない
- FR-9: アクティブ会話の最新メッセージが選択肢つきアシスタント発言である間、テキスト入力欄と送信ボタンを無効化する。選択肢のない応答（内容入力の促し、完了、中断、通常チャット応答）では入力欄は有効とする。会話切り替え時もアクティブ会話の最新メッセージに基づいて判定する
- FR-10: 最新のアシスタント発言以外に含まれる選択肢ボタンは無効化して表示する（履歴として何が提示されたかは残す）
- FR-11: ウィザード途中・完了後の会話は他の会話と同様に履歴一覧に残り、切り替えて戻ると選択肢を含めて再表示され（有効・無効は FR-9/10 の規則どおり）、途中のフローを継続できる
- FR-12: フロー用のユーザー発言（開始トリガー文、「カテゴリ: 請求」等の定型文、累積形式）が既存のレポートシナリオ（「担当」「カテゴリ」「種別」キーワード）に誤マッチしないよう、Mockの判定はフロー系を優先する。既存の定型レポート・手入力キーワード応答・非マッチ案内の挙動は変えない
- FR-13: 選択肢コンポーネントの検証（必須フィールド・選択肢1件以上）を `chatApi.ts` に追加し、不正な場合は既存方針どおり「応答を表示できませんでした」のエラー吹き出しにフォールバックする

## 画面・UIフロー

既存のチャット画面構成（Sidebar ＋ ChatWindow ＋ MessageInput）に新規画面は追加しない。変更は Sidebar のボタン追加と、吹き出し内の選択肢描画・入力欄の無効化制御。

```
ユーザー: （サイドバーの[新規問い合わせ]を押下 → 新規会話作成）
ユーザー: 新規問い合わせ                        ← 開始トリガー文（吹き出し表示）
ボット:   問い合わせのカテゴリを選んでください
          [請求] [技術] [アカウント] [その他] [キャンセル]   ← 入力欄は無効
ユーザー: カテゴリ: 請求                         ← ボタンクリックで送信
ボット:   緊急度を選んでください [高] [中] [低] [キャンセル]
ユーザー: 緊急度: 高
ボット:   問い合わせ内容を入力してください        ← 選択肢なし＝入力欄が有効化
ユーザー: カテゴリ: 請求 / 緊急度: 高 / 内容: 請求額が二重になっている
ボット:   （要約表: カテゴリ/緊急度/内容）この内容で登録しますか？
          [登録する] [やり直す] [キャンセル]
ユーザー: 登録する
ボット:   問い合わせを受け付けました。受付番号: INQ-XXXX
```

- キャンセル押下: 「問い合わせの登録を中止しました」等の中断メッセージで終了し、通常チャットに戻る
- やり直す押下: カテゴリ選択の質問が再提示され、フローを最初から進める
- 過去のステップの選択肢ボタンは無効化表示。押下できるのは常に最新のアシスタント発言内の選択肢のみ

## 受け入れ基準

- AC-1: サイドバー上部の New Chat 直下に「新規問い合わせ」ボタンが表示され、押下すると新規会話が作成されてカテゴリ選択の質問（4カテゴリ＋キャンセルの選択ボタン）が表示される（FR-1, 2, 4, 5。検証方法: 手動確認）
- AC-2: カテゴリ→緊急度を選択し、内容を入力し、確認で「登録する」を選ぶと、要約表を経て受付番号つきの完了メッセージが表示される（FR-5, 6, 8。検証方法: 手動確認）
- AC-3: 選択肢提示中はテキスト入力欄・送信ボタンが無効で、内容入力ステップに進むと有効になる（FR-9。検証方法: 手動確認）
- AC-4: 回答済みの過去の選択肢ボタンは無効化表示され、押下しても何も送信されない（FR-10。検証方法: 手動確認）
- AC-5: 各選択ステップで「キャンセル」を押すと中断メッセージが表示され、その会話で通常のチャット（例: 手入力の担当者別質問）が継続できる（FR-7。検証方法: 手動確認）
- AC-6: 確認ステップで「やり直す」を選ぶとカテゴリ選択から再開できる（FR-8。検証方法: 手動確認）
- AC-7: `POST /api/chat` の統合テストで、(a) 開始トリガー文→カテゴリ質問＋選択肢コンポーネント、(b) 累積形式の内容送信→要約表＋確認選択肢、(c) 「登録する」→受付番号を含む完了応答、が契約どおり返る（FR-2, 3, 5, 6, 8。検証方法: 自動テスト（MockMvc））
- AC-8: フロー用定型文（「カテゴリ: 請求」等）に対して既存のカテゴリ別レポート応答（表＋棒グラフ）が返らない（FR-12。検証方法: 自動テスト）
- AC-9: 既存挙動の退行がない: 定型レポートボタン、手入力の「担当」「カテゴリ」キーワード応答、非マッチ時の案内文、New Chat・会話切り替え（FR-12。検証方法: 既存自動テスト＋手動確認）
- AC-10: ウィザード途中で別会話に切り替えて戻ると、選択肢を含む履歴が再表示され、最新の選択肢からフローを継続できる（FR-11。検証方法: 手動確認）
- AC-11: 選択肢が0件など不正な選択肢コンポーネントを含む応答はエラー文言の吹き出しにフォールバックし、アプリはクラッシュしない（FR-13。検証方法: 自動テスト（chatApi検証の単体）または手動確認）
- AC-12: [chat-response-schema.md](../chat-response-schema.md) に選択肢コンポーネントの定義とサンプルが追記され、AC-7 のテストが検証する実応答と一致している（FR-3。検証方法: 手動確認）
- AC-13: 応答待ち中は「新規問い合わせ」ボタンが無効化され、連打しても会話が1つしか作られない（FR-1。検証方法: 手動確認）

## 制約

- 技術スタックは既存踏襲: フロント Vue 3 + TypeScript + Pinia（Vite）、バックエンド Java Spring Boot + Maven
- バックエンドはステートレスを維持する（会話状態・セッションを持たない）。フローの進行状態は「直前のユーザー発言の定型文」だけで判別できる設計とし、`ChatRequest`（`message` のみ）は変更しない
- 選択肢コンポーネントは既存の `UiComponent` 判別共用体（`@JsonTypeInfo`/`@JsonSubTypes`）の拡張として追加し、[chat-response-schema.md](../chat-response-schema.md) の「将来の拡張」手順に従う
- 応答スペック内の文字列は既存方針どおり `v-html` を用いずテキストとして描画する（XSS防止）
- Mockのキーワード判定はフロー系定型文を既存レポートキーワード（「担当」「カテゴリ」「種別」）より先に評価する
- スタイルは既存のCSSカスタムプロパティと既存トーンに合わせ、過度な作り込みをせず「動くプロトタイプ」を優先する既存方針を踏襲する

### 前提（調査により確認）

- 開始トリガー文は「担当」「カテゴリ」「種別」を含まない20文字以内の定型文（例「新規問い合わせ」）とし、既存タイトル規則によりそのまま会話タイトルになる
- 選択肢クリックで送信する定型文の具体的な文言・累積形式の区切り文字は、Mockが一意に判別できることを条件に設計フェーズで決定する
- 受付番号は架空でよい（固定値かランダムかは設計フェーズの判断。保存しないため一意性の保証は不要）
- 受付完了メッセージでは入力内容の要約を再掲しない（確認ステップで表示済み）
- 緊急度（高/中/低）は既存モックデータにない新規項目だが、登録データは保存しないため既存のレポートシナリオ（担当者別・カテゴリ別）への影響はない
- 空文字・空白のみの内容は既存の送信抑止（フロント `canSend`／バックエンド `@NotBlank`）により送信されない。内容の文字数上限は設けない
- 利用者は開発者本人およびデモ閲覧者のみで、認証・ロール分岐なし。会話・登録データの永続化なし、既存データの移行不要（既存方針踏襲）
- 1つの応答に含まれる選択肢コンポーネントは1個を想定する（複数併記の要件はない）

## 重大な非機能要件

- **表示の安全性（XSS防止）**: 選択肢のラベル等、応答スペック内の文字列は将来LLM由来になる前提で、HTMLとして解釈せず必ずテキストとして描画する（既存方針の踏襲）
- **不正応答への耐性**: 不正な選択肢コンポーネントでフロントがクラッシュせずエラー表示にフォールバックする（FR-13。既存方針の踏襲）

## 決定記録

- Q1: 「新規問い合わせ」の意味 → 問い合わせチケットの新規登録ウィザード（理由: 本アプリのデモ題材が問い合わせ管理業務であり、集計系は既存の定型レポートボタンが担当済み）
- Q2: 対話フローの制御場所 → バックエンドMockが選択肢つき応答を返して駆動（理由: 「Mockを将来LLMに差し替えるだけ」という本リポジトリの設計思想に合致し、対話の主導権もLLM側に置ける）
- Q3: ボタン配置 → サイドバー上部・New Chat直下（理由: 「新しい会話を始める」操作同士が隣接し迷わない）
- Q4: 会話の扱い → 常に新規会話を作成して開始（理由: 既存の定型レポートボタンと同じパターンで実装でき、進行中のチャットを汚さない）
- Q5: ステップ構成 → カテゴリ→緊急度→内容（自由入力）→確認（理由: 「なるべく選択式」の意図に沿い選択ステップを2つ確保しつつ、問い合わせ内容だけは定型化できないため自由入力を残す）
- Q6: 選択肢提示中のテキスト入力欄 → 無効化（理由: 「なるべく選択式」の意図に沿い、ステートレスMockではフロー途中の想定外テキストで対話が崩れる経路をUI側で防ぐ）
- Q7: 途中キャンセル → 各選択ステップにキャンセル選択肢（理由: 入力欄無効化と組み合わせる場合、キャンセル選択肢がないとフローから抜け出せない）
- Q8: 登録データ → 保存しない・受付完了表示のみ（理由: バックエンドのステートレス維持というモック方針に合致）
- Q9: フロー状態の運び方 → 内容送信時にフロントが全回答を累積した定型文で送信（理由: リクエストAPIを変えずに確認ステップの要約表示まで実現でき、ステートレスを維持。送信文が吹き出しに見えるのもデモとして透明）
- Q10: 回答済みの選択肢ボタン → 最新以外は無効化（理由: 過去の選択肢の押下でフローが巻き戻る混乱を防ぎ、提示履歴は残す）
- Q11: 確認の「やり直す」 → カテゴリ選択から最初からやり直し（理由: 項目単位修正はステートレスMockでは分岐が重く、全3ステップなので最初からでも負担が小さい）

## 用語集

- **新規問い合わせ**: 本書では「問い合わせ管理業務における問い合わせチケットの新規登録」を指す。「LLMへの問い合わせ（質問）」の意味ではない（後者は本フローの裏でMockが担う部分）
- **ウィザード／フロー**: アシスタントが質問と選択肢を順に提示し、ユーザーの回答で次のステップへ進む一連の対話。実体はステートレスなMockが「直前のユーザー発言の定型文」を判別して次の応答を返す仕組み
- **選択肢コンポーネント**: 応答JSONの `components` に新設する描画指示。アシスタント吹き出し内の選択ボタン群としてレンダリングされ、クリックが定型文のユーザー発言送信になる
- **登録**: 受付完了メッセージを返すことのみを指し、データはどこにも保存されない（モック）
