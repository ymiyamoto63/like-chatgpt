# メッセージ入力補完（ゴーストテキスト）

## 概要

メッセージ入力欄で、入力途中のテキストから後続の文を推測し、入力欄内にゴーストテキスト（薄色の続き文字）として表示する。Tabキーで確定すると入力欄に全文が反映される。候補の推測はバックエンドに新設するサジェストAPI（モック実装）が行い、既存レポートシナリオの定型文への前方一致で残りの文字列を返す。

## 背景・目的

自由入力の手間を減らし、モックが応答できる定型文へユーザーを自然に誘導する。既存方針（対話ロジックをバックエンド＝将来のLLM側に置き、モックで構図を検証する）に合わせ、推測ロジックもバックエンドAPIとして確立し、将来LLMベースの補完に差し替えられる形にする。

## スコープ

- フロントエンド: [MessageInput.vue](../../frontend/src/components/MessageInput.vue) にゴーストテキスト表示・Tab確定・Esc消去・デバウンス付き候補取得を追加
- フロントエンド: サジェストAPIクライアントの追加（[chatApi.ts](../../frontend/src/api/chatApi.ts) と同様の検証パターンで新規モジュールまたは同ファイルに追加）
- バックエンド: サジェストAPIエンドポイントの新設（例: `POST /api/suggest`。パスは設計フェーズで確定）とコントローラ・サービス・DTOの追加
- バックエンド: 定型文リスト（レポート系）への前方一致で候補1件を返すモックロジックとテスト
- ドキュメント: サジェストAPIの契約を docs/ に追記（[chat-response-schema.md](../chat-response-schema.md) の流儀に準拠）

## 対象外（Non-goals）

- 実際のLLMによる補完（モックのみ。APIの形は差し替えを見据える）
- 複数候補のドロップダウン表示・候補切り替えUI（APIも1件のみ返す）
- 新規問い合わせウィザードの定型文・モニタリングアラート定型文の補完（ボタン・選択肢経由で送られるため対象外）
- 補完の学習・履歴に基づくパーソナライズ
- 送信そのものの自動化（Tab確定後の送信は従来どおりユーザーのEnter/送信ボタン操作）

## 影響範囲

- **フロントエンド 画面コンポーネント** — [MessageInput.vue](../../frontend/src/components/MessageInput.vue)（ゴーストテキスト描画、キーハンドリング、IME composition 判定、デバウンス）
- **フロントエンド APIクライアント・型定義** — サジェストAPIクライアントの新設（[types/chat.ts](../../frontend/src/types/chat.ts) 相当の型追加を含む）
- **フロントエンド 状態管理** — 変更なしの想定（補完状態は MessageInput.vue ローカルで完結。設計で覆る場合あり）
- **バックエンド API** — コントローラ・リクエスト/レスポンスDTOの新設（[ChatController.java](../../backend/src/main/java/com/example/chatbackend/ChatController.java) と同パターン）
- **バックエンド サービスロジック** — サジェストサービスの新設（[MockChatService.java](../../backend/src/main/java/com/example/chatbackend/MockChatService.java) と同様のキーワード定数管理）＋テスト
- **ドキュメント** — サジェストAPI契約の追記
- DBスキーマへの影響なし（ステートレス維持）

## 機能要件

- FR-1: 入力欄のテキストが1文字以上のとき、入力停止から約300msのデバウンス後にサジェストAPIへ現在の入力テキストを送信し、候補を取得する。
- FR-2: バックエンドは定型文リスト（レポート系のみ）に対して入力テキストの前方一致を判定し、一致した場合は定義順で最初の1件について「残りの文字列」を返す。一致がない場合は候補なしを返す。
- FR-3: 候補が得られたとき、入力欄内のカーソル位置以降にゴーストテキスト（薄色）として残り文字列を表示する。ゴーストテキストは送信対象のテキストには含まれない。
- FR-4: ゴーストテキスト表示中に Tab キーを押すと補完が確定し、入力欄のテキストが定型文全文になる。カーソルは末尾に移動し、ゴーストテキストは消える。Tab の既定動作（フォーカス移動）は抑止する。ゴーストテキスト非表示時の Tab は従来どおり。
- FR-5: ゴーストテキスト表示中に Esc キーを押すと候補を消す。その後さらに入力が変化すれば再度候補を取得・表示する。
- FR-6: IME 変換中（composition 中）は候補の取得・表示を行わず、変換確定後のテキストで FR-1 の流れを開始する。
- FR-7: 入力テキストが空になったとき、入力が定型文と一致しなくなったとき、および入力欄が無効化されている間（選択肢ウィザード中・応答待ち中）は、候補を表示しない（表示中なら消す）。
- FR-8: 入力テキストが定型文全文と完全一致しているときは候補を表示しない（残りが空のため）。
- FR-9: サジェストAPIの呼び出しが失敗した場合は候補を表示せず、エラー表示は行わない（補完は補助機能であり本文入力・送信を妨げない）。
- FR-10: デバウンス後に複数の応答が前後して届いた場合でも、最新の入力テキストに対応する候補のみを表示する（古い応答は破棄）。

## 画面・UIフロー

既存のチャット画面の入力欄（MessageInput.vue）のみに閉じる。新規画面・遷移なし。

1. ユーザーが入力欄にタイプする（例: 「当月担」）。
2. 300ms 入力が止まると候補を取得し、続き（例: 「当者別問い合わせ」の部分）が薄色で入力欄内に表示される。
3. Tab で全文「当月担当者別問い合わせ」が入力欄に入る。Esc または続けての入力変化で候補は消える/更新される。
4. 送信は従来どおり Enter または送信ボタン。

## 受け入れ基準

- AC-1: 定型文の先頭部分（例: 「当月担」）を入力して300ms待つと、残りの文字列がゴーストテキストとして表示される。（検証方法: 手動確認）
- AC-2: ゴーストテキスト表示中に Tab を押すと入力欄が定型文全文になり、Enter で従来どおり送信できる。（検証方法: 手動確認）
- AC-3: ゴーストテキスト表示中に Esc を押すと候補が消え、入力を変化させると再表示される。（検証方法: 手動確認）
- AC-4: IME 変換中は候補が表示されない。（検証方法: 手動確認）
- AC-5: どの定型文にも前方一致しない入力では候補が表示されない。（検証方法: 手動確認）
- AC-6: サジェストAPIが、定型文の先頭部分に対して残り文字列を、非一致・完全一致・空文字に対して候補なしを返す。（検証方法: 自動テスト — バックエンド単体テスト）
- AC-7: バックエンド停止状態でも入力・送信操作がエラー表示なく行える（補完だけが出ない）。（検証方法: 手動確認）
- AC-8: 入力欄が無効→有効に戻る既存挙動（フォーカス復帰）が維持される。（検証方法: 手動確認）

## 制約

- バックエンドはステートレスなモック実装を維持する（Spring Boot、[ChatController.java](../../backend/src/main/java/com/example/chatbackend/ChatController.java) と同じ REST パターン、`@Valid` によるリクエスト検証）。
- フロントは Vue 3 `<script setup>` + TypeScript + Tailwind。API 応答はフロント側で [chatApi.ts](../../frontend/src/api/chatApi.ts) と同様に型検証してから使う。
- CORS 設定（WebConfig.java）に新エンドポイントが含まれるようにする。
- 言語をまたぐ定数（定型文の文言）は、既存の monitoringAlert 定数と同様「一致させること」のコメントで対応する（自動同期はしない）。

## 重大な非機能要件

なし（デバウンスによる呼び出し抑制は FR-1 で規定済み。個人情報・認証は本プロジェクトのスコープ外）。

## 未決定事項

- 補完対象の定型文の正確な文言 — 選択肢: 既存レポートボタン文言（「当月担当者別問い合わせ」「当月カテゴリ別問い合わせ」「直近14日間の日別問い合わせ」）をそのまま採用 / 補完専用の自然文（例: 「今月の問い合わせ件数を担当者別にまとめて」）を別途定義 / 推奨仮置き: 既存レポートボタン文言をそのまま採用（[reportButtons.ts](../../frontend/src/constants/reportButtons.ts) に既にあり、キーワードマッチで確実に応答が返ることが保証されるため）/ 外れた場合の影響: バックエンドの定型文リストの文字列定数を差し替えるだけで済み、構造への影響はない。
- サジェストAPIのパス・リクエスト形式（`POST /api/suggest` に `{ text }` を想定）— 選択肢: POST+JSON（既存 /api/chat と同型）/ GET+クエリ / 推奨仮置き: POST+JSON（既存パターンとの一貫性）/ 外れた場合の影響: クライアントとコントローラの薄い層の変更のみ。

## 決定記録

- Q1: 補完のUI形態 → ゴーストテキスト（理由: 「入力内容から後続を推測して補完」という要望に最も忠実で、Smart Compose 風の自然な UX）
- Q2: 候補の生成元 → バックエンド新API（理由: 推測ロジックをバックエンド＝将来のLLM側に置く既存方針と一致）
- Q3: 確定操作 → Tab で確定・Esc で消去（理由: 既存の Enter 送信仕様と衝突せず、Copilot 等の慣習に一致）
- Q4: 推測ロジック → 定型文への前方一致で1件（理由: 既存のキーワードマッチ型モックと同じ発想で実装が安全）
- Q5: 取得タイミング → 1文字目から・約300msデバウンス（理由: 連打リクエスト回避、標準的実装）
- Q6: IME対応 → 変換確定後のみ補完（理由: 未確定文字列での誤補完と Tab/IME 操作の衝突を回避）
- Q7: 複数一致時 → 定義順の先頭1件のみ返す（理由: ゴーストテキストは構造上1件しか出せないため API も1件で単純化）
- Q8: 定型文の範囲 → レポート系のみ（理由: ウィザード・アラート定型文はボタン/選択肢経由で送られるため手入力補完の対象外とするのが既存UIと整合）

## 用語集

- 補完（ゴーストテキスト）: 入力欄内に薄色で表示される「送信テキストには含まれない」候補の残り文字列。Tab 確定で初めて実テキストになる。
- 定型文: バックエンドのキーワードマッチで確実に定型応答が返る、あらかじめ定義された依頼文。本機能ではレポート系のみを指す。
- サジェストAPI: 入力途中テキストを受け取り補完候補（残り文字列）を返す新設エンドポイント。モック実装だが将来 LLM に差し替える前提の契約とする。
