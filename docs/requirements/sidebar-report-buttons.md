# サイドバー定型レポートボタン

## 概要

左サイドバーを上下2分割し、上部は現状どおり New Chat ボタンと過去チャット履歴の一覧、下部に定型レポート用のプロンプト実行ボタン群（「当月担当者別問い合わせ」「当月カテゴリ別問い合わせ」の2つ）を配置する。ボタン押下で対応する定型プロンプト文が新規会話のユーザー発言として送信され、既存のチャット応答フロー（表＋棒グラフのモック応答）がそのまま実行される。

## 背景・目的

よく使う集計レポート（担当者別・カテゴリ別の問い合わせ件数）を、毎回自然文を入力しなくてもワンクリックで呼び出せるようにする。既存のキーワードマッチによるモック応答（表＋棒グラフ）のデモ導線を明示的なボタンとして提示することで、初見の利用者にも「何を聞けば何が返るか」が分かるようにする狙い。

## スコープ

- フロントエンド: [Sidebar.vue](../../frontend/src/components/Sidebar.vue) を上下2分割レイアウトに改修
  - 上部: New Chat ボタン＋会話履歴一覧（現状の機能・挙動を維持）
  - 下部: 定型レポートボタン2つ（「当月担当者別問い合わせ」「当月カテゴリ別問い合わせ」）
- フロントエンド: ボタン押下時に「新規会話を作成 → ボタンラベル文字列をユーザー発言として送信」する処理（既存ストアアクションの組み合わせで実現）
- 定型レポートボタンの定義（ラベル＝送信文）はフロントエンド内の固定定義とする

## 対象外（Non-goals）

- バックエンドの変更（既存のキーワードマッチ応答をそのまま利用する）
- レポート専用APIの新設
- ボタン定義のサーバー管理・動的取得
- 「当月」の実日付反映（応答は既存モックの固定架空データのまま。実際の月での集計は将来の実データ接続フェーズの課題）
- ボタンの追加・編集UI（ボタンの増減はコード修正で対応）
- 2つ以外のレポートボタン（今回は例示の2つで確定）

## 影響範囲

- **フロントエンド 画面コンポーネント** — [Sidebar.vue](../../frontend/src/components/Sidebar.vue): 2分割レイアウトとレポートボタン群の追加が本改修の中心
- **フロントエンド 状態管理** — [conversationStore.ts](../../frontend/src/stores/conversationStore.ts): 既存の `createConversation()` と `sendMessage()` の組み合わせで実現できる見込み。新規アクションを足すかどうかは設計フェーズの判断
- バックエンドAPI・サービスロジック・型定義・APIクライアントへの影響: なし

## 機能要件

- FR-1: 左サイドバーは上下2つのセクションに分割される。上部セクションには現状どおり New Chat ボタンと会話履歴一覧を表示し、既存の機能・挙動（新規作成、会話選択、アクティブ表示）を維持する
- FR-2: 下部セクションには定型レポートボタンとして「当月担当者別問い合わせ」「当月カテゴリ別問い合わせ」の2つを表示する
- FR-3: 定型レポートボタンを押下すると、常に新規会話が作成・アクティブ化され、ボタンラベルと同一の文字列がユーザー発言としてその会話に送信される（既存の `POST /api/chat` 経由。手入力送信と同じフロー）
- FR-4: 送信されたユーザー発言は通常のチャットと同様に吹き出しとして表示され、既存のキーワードマッチにより「当月担当者別問い合わせ」→担当者別シナリオ（表＋棒グラフ）、「当月カテゴリ別問い合わせ」→カテゴリ別シナリオ（表＋棒グラフ）の応答が表示される
- FR-5: レポート実行で作成された会話は既存ルール（先頭メッセージの先頭20文字）によりボタンラベルがタイトルとなって履歴一覧に追加され、他の会話と同様に選択・再表示できる
- FR-6: 応答待ち中（`isLoading` 中）は定型レポートボタンを無効化し、多重送信を防ぐ（既存の送信ボタンと同様の扱い）
- FR-7: 定型レポートボタンの定義（ラベル＝送信文）はフロントエンド内の固定定義とし、バックエンドには持たせない

## 画面・UIフロー

既存のチャット画面構成（Sidebar ＋ ChatWindow ＋ MessageInput）のうち Sidebar のみ変更。新規画面は追加しない。

```
┌ Sidebar ────────────┐
│ [ New Chat ]         │  ← 上部: 現状どおり
│ 会話履歴1            │     （履歴が増えたらこの領域内でスクロール）
│ 会話履歴2            │
│ ...                  │
├──────────────────┤  ← セクション境界
│ 定型レポート          │  ← 下部: 下端に固定
│ [当月担当者別問い合わせ] │
│ [当月カテゴリ別問い合わせ]│
└─────────────────┘
```

1. ユーザーが下部の定型レポートボタンを押下
2. 新規会話が作成されてアクティブになり、ボタンラベル文がユーザー吹き出しとして表示される
3. 既存のローディング表示を経て、アシスタント吹き出しに要約テキスト＋表＋棒グラフが表示される
4. 会話は履歴一覧の先頭に追加され、以降は通常の会話として継続入力・切り替えが可能

## 受け入れ基準

- AC-1: サイドバーが上下2分割され、上部に New Chat と会話履歴、下部に「当月担当者別問い合わせ」「当月カテゴリ別問い合わせ」の2ボタンが表示される（FR-1, 2。検証方法: 手動確認）
- AC-2: 「当月担当者別問い合わせ」ボタンを押すと、新規会話が作成され、同名のユーザー吹き出しに続いて担当者別件数の表＋棒グラフの応答が表示される（FR-3, 4。検証方法: 手動確認）
- AC-3: 「当月カテゴリ別問い合わせ」ボタンを押すと、カテゴリ別件数の表＋棒グラフの応答が表示される（FR-4。検証方法: 手動確認）
- AC-4: 既存会話でメッセージをやり取りした状態でレポートボタンを押しても、既存会話にはメッセージが追加されず、新規会話側に送信される（FR-3。検証方法: 自動テスト（ストア単体）または手動確認）
- AC-5: レポート実行後、履歴一覧の先頭にボタンラベルをタイトルとする会話が追加され、他会話へ切り替えて戻ると表＋グラフが再表示される（FR-5。検証方法: 手動確認）
- AC-6: 応答待ち中は定型レポートボタンが無効化され、連打しても会話が1つしか作られない（FR-6。検証方法: 手動確認）
- AC-7: New Chat・会話選択・手入力送信の既存挙動が改修後も変わらない（FR-1。検証方法: 手動確認）
- AC-8: 会話履歴が多数ある場合、スクロールするのは上部の履歴領域のみで、下部のレポートボタンは常に表示され続ける（FR-1, 2。検証方法: 手動確認）

## 制約

- 技術スタックは既存踏襲: フロント Vue 3 + TypeScript + Pinia（Vite）、バックエンド Java Spring Boot + Maven
- バックエンドは変更しない（ステートレス・キーワードマッチのモックを維持）
- 送信は既存の `conversationStore` のアクション経由とし、APIクライアント（`chatApi.ts`）・レスポンススキーマ（`docs/chat-response-schema.md`）には手を入れない
- スタイルは既存の CSS カスタムプロパティ（`--border`, `--bg`, `--accent` 等）と Sidebar の既存トーンに合わせる
- 過度な作り込みをせず「動くプロトタイプ」を優先する既存方針を踏襲する

### 前提（調査により確認）

- ボタンラベル「当月担当者別問い合わせ」は既存キーワード「担当」に、「当月カテゴリ別問い合わせ」は「カテゴリ」にマッチするため、ラベルをそのまま送信すれば既存モックが意図した応答を返す（[MockChatService.java](../../backend/src/main/java/com/example/chatbackend/MockChatService.java) で確認）
- サイドバーの2分割は「上部＝可変高さ＋履歴スクロール、下部＝下端固定」のレイアウトとする（メモの「2分割」の解釈。下部には「定型レポート」等の小見出しを付けてよい）
- 会話タイトルは既存ロジック（先頭メッセージの先頭20文字）に任せる。ボタンラベルは20文字以内のためそのまま全文がタイトルになる
- 利用者は開発者本人およびデモ閲覧者のみで、認証・ロール分岐なし（既存方針踏襲）
- 会話は永続化されないため既存データの移行は不要（既存方針踏襲）

## 決定記録

- Q1: ボタン押下時の動作方式 → 定型文をユーザー発言として送信（理由: 既存の sendMessage フローにそのまま乗り、バックエンド変更不要で実装コスト最小。何を聞いたかが吹き出しに残る）
- Q2: 送信先の会話 → 常に新規会話を作成（理由: レポートが独立した履歴エントリとして残り、進行中のチャットを汚さない）
- Q3: ボタン定義の持ち方 → フロントエンドに固定定義（理由: 「動くプロトタイプ優先」の既存方針。バックエンドのステートレスを維持）
- Q4: ボタンの数 → メモの例の2つで確定（理由: 既存モックの2シナリオと一対一に対応し、バックエンド変更なしで全ボタンが意味のある応答を返せる）
- Q5: 送信するプロンプト文 → ボタンラベルをそのまま送信（理由: 実装最小。会話タイトル＝ボタン名になり履歴で判別しやすい。既存キーワードにもマッチ）
- Q6: 「当月」の扱い → モック固定応答のまま、バックエンド変更なし（理由: 実データ非接続のモック方針を踏襲。実際の月での集計は実データ接続フェーズの課題）

## 用語集

- **定型レポート**: 本書では「事前定義された固定のプロンプト文をワンクリックで送信するショートカット」を指す。帳票出力・ファイル生成の類ではなく、応答は通常のチャット吹き出し（テキスト＋表＋棒グラフ）として表示される
- **当月**: ボタンラベル上の表現であり、システム上は実際の年月での絞り込みは行わない（応答はモックの固定架空データ）
