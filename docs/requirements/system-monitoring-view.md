# システム構成図モニタリング画面

## 概要

サイドバーから切り替えられる専用モニタリング画面を新設し、モックのシステム構成図（3層Web構成・8ノード程度）をSVGで描画する。各ノード（機器）のCPU・メモリ使用率と、ノード間接続（エッジ）の帯域使用率を、3段階の色分け（正常/警戒/危険）＋ゲージ＋数値でグラフィカルに表示する。画面表示中は5秒間隔でバックエンドの新APIをポーリングし、ページリロードなしにリアクティブに再描画する。

## 背景・目的

チャットUI試作プロジェクトに、システム運用監視ダッシュボードの試作を追加する。コンポーネント間のネットワークボトルネックや機器リソースの逼迫箇所を一目で把握できる可視化表現と、ライブ更新のUXを検証することが狙い。実監視システムとの接続は行わず、モックデータで表現力・更新機構を確認する。

## スコープ

- サイドバーにモニタリング画面への切替ボタンを追加し、メイン領域をチャット画面⇄モニタリング画面で切り替えられるようにする
- バックエンドに新規エンドポイント（GET）を追加し、システム構成（トポロジー）とその時点のメトリクススナップショットを返す
- モック構成: Internet → LB → Web×2 → App×2 → Cache / DB（Primary・Replica）程度の3層Web構成・8ノード規模
- 構成図の自作SVG描画（固定座標レイアウト、表示専用）
- ノード: CPU/メモリのミニゲージ＋使用率数値＋最悪値に基づく3段階色分け
- エッジ: 帯域使用率ラベル＋3段階色分け
- 5秒間隔のRESTポーリングとリアクティブ再描画、最終更新時刻の表示
- ポーリング失敗時のエラーバナー表示と自動復帰
- モックメトリクスの変動生成（ランダムウォーク＋周期的スパイク）

## 対象外（Non-goals）

- 実監視システム・実機メトリクスとの接続
- メトリクス履歴の保存・時系列グラフ表示
- アラート通知（メール・プッシュ等）や警戒/危険の一覧サマリー表示
- ノードのクリック・ホバー等のインタラクション（詳細パネル、ツールチップ含む）
- トポロジーの編集・動的な構成変更のシミュレーション
- ブラウザタブ非表示時のポーリング停止（visibilitychange対応）
- SSE / WebSocket によるプッシュ配信
- 認証・認可、データ永続化（プロジェクト全体の方針どおり）

## 影響範囲

- **フロントエンド 画面コンポーネント**: 新規モニタリング画面コンポーネント（構成図SVG、ノード、エッジ、エラーバナー）、`App.vue`（メイン領域の画面切替）、`Sidebar.vue`（切替ボタン追加）
- **フロントエンド 状態管理**: 画面切替状態と、モニタリングデータ・ポーリング制御を持つ新規Piniaストア（既存 `conversationStore` は変更不要の見込み）
- **フロントエンド APIクライアント**: 新APIのクライアント関数と応答検証（`chatApi.ts` と同じ検証方針の新ファイルまたは追記）
- **バックエンド**: 新規コントローラ／サービス（トポロジー定義とメトリクス生成）。既存の `POST /api/chat` 系（`ChatController` / `MockChatService` / `UiComponent` スキーマ）には手を入れない
- **ドキュメント**: 新APIの応答スキーマを文書化（`docs/api/chat-response-schema.md` とは別ファイルまたは別節）

## 機能要件

- FR-1: サイドバーにモニタリング画面への切替ボタンを表示する。押下するとメイン領域がチャット画面からモニタリング画面に切り替わる。チャットの会話状態は切替によって失われない。
- FR-2: チャット画面へ戻る導線を提供する（サイドバーの既存操作＝New Chat／会話選択／定型レポート／新規問い合わせのいずれかでチャット画面に復帰する）。
- FR-3: バックエンドは新規GETエンドポイントで、固定トポロジー（8ノード程度の3層Web構成：Internet、LB、Web×2、App×2、Cache、DB Primary/Replica を想定）と、各ノードのCPU使用率・メモリ使用率（0〜100の数値）、各エッジの帯域使用率（0〜100の数値）を1レスポンスで返す。
- FR-4: モニタリング画面は取得したトポロジーを自作SVG（固定座標レイアウト）で描画する。外部の描画ライブラリは導入しない。
- FR-5: 各ノードは、CPU・メモリそれぞれのミニゲージバーと使用率数値（%）を表示し、CPU・メモリの最悪値に応じてノード枠を3段階に色分けする（70%未満=正常/緑、70%以上=警戒/黄、90%以上=危険/赤）。
- FR-6: 各エッジは、帯域使用率の数値ラベル（%）を表示し、ノードと同じ閾値（70%/90%）で線を3段階に色分けする。
- FR-7: モニタリング画面の表示開始時に即座に初回データを取得し、以降5秒間隔でポーリングして取得のたびに画面リフレッシュ（ページリロード）なしに再描画する。チャット画面へ戻ったらポーリングを停止する。
- FR-8: 画面に最終更新時刻（データ取得に成功した時刻、HH:mm:ss）を表示する。
- FR-9: ポーリングが失敗した場合（バックエンド停止・ネットワークエラー・応答検証エラー）、直前の取得データによる図の表示を維持したまま接続エラーのバナーを表示する。ポーリングは継続し、成功したらバナーを消して自動復帰する。初回取得から失敗した場合は図の代わりにエラー表示を出す。
- FR-10: フロントエンドはAPIクライアント層で応答の構造を検証し、必須フィールドの欠落・型不一致がある応答は失敗（FR-9のエラー扱い）とする。未知の追加フィールドは無視する（既存 `chatApi.ts` と同方針）。
- FR-11: バックエンドのメトリクス生成は、ノード・エッジごとのベースラインを基準に前回値から少しずつ変動するランダムウォークとし、加えて一定周期でいずれかのノードまたはエッジが警戒・危険域（70%超〜90%超）まで上昇して戻るスパイクを発生させる。デモ中に色の3段階変化が必ず観察できること。

## 画面・UIフロー

- 既存レイアウト（ヘッダ＋サイドバー＋メイン領域）は維持する。サイドバーの上部ボタン群（New Chat・新規問い合わせの並び）に「システムモニタリング」ボタンを追加する。
- 「システムモニタリング」押下 → メイン領域が `ChatWindow` からモニタリング画面に切り替わる。ボタンは選択中であることが分かる表示にする。
- モニタリング画面の構成: タイトル、最終更新時刻、（エラー時のみ）接続エラーバナー、構成図SVG。それ以外の要素は置かない。
- サイドバーでNew Chat／会話選択／定型レポート／新規問い合わせを操作 → メイン領域がチャット画面に戻り、ポーリングが停止する。
- 既存のダークモード対応（Tailwindの `dark:` バリアント）に追随し、ライト/ダーク両テーマで3段階色が判別できること。

## 受け入れ基準

- AC-1: 新規GETエンドポイントを呼ぶと、8ノード程度のトポロジーと全ノードのCPU/メモリ・全エッジの帯域使用率（いずれも0〜100）を含むJSONが返る（FR-3）。（検証方法: 自動テスト）
- AC-2: 同エンドポイントを連続で呼ぶと値が前回から変動しており、値は常に0〜100の範囲に収まる（FR-11）。（検証方法: 自動テスト）
- AC-3: サイドバーのボタンでモニタリング画面に切り替わり、構成図・ゲージ・数値が表示される。チャットに戻ると元の会話がそのまま表示される（FR-1, FR-2, FR-4, FR-5）。（検証方法: 手動確認）
- AC-4: モニタリング画面を開いたまま観察すると、5秒ごとに数値・ゲージがページリロードなしに更新され、最終更新時刻が進む（FR-7, FR-8）。（検証方法: 手動確認）
- AC-5: 数分観察すると、いずれかのノードまたはエッジが黄（70%以上）や赤（90%以上）に変化し、その後正常色に戻るスパイクが確認できる（FR-6, FR-11）。（検証方法: 手動確認）
- AC-6: モニタリング画面表示中にバックエンドを停止すると、図は直前データのまま接続エラーバナーが表示され、バックエンドを再起動すると次のポーリング成功時にバナーが消えて更新が再開する（FR-9）。（検証方法: 手動確認）
- AC-7: チャット画面表示中は新APIへのリクエストが発生しない（開発者ツールのネットワークタブで確認）（FR-7）。（検証方法: 手動確認）
- AC-8: 応答JSONの必須フィールドを欠落させたモック応答に対して、フロントエンドがクラッシュせずエラー扱い（バナー/エラー表示）となる（FR-10, FR-9）。（検証方法: 手動確認）

## 制約

- フロントエンド: Vue 3（Composition API）＋ Pinia ＋ Tailwind CSS 4。ルーターは未導入のため、画面切替はコンポーネント条件描画＋ストア状態で行う（ルーター新規導入はしない）。
- 描画ライブラリ・チャートライブラリは導入しない（既存の棒グラフ・トレンドグラフも自作であることに合わせる）。
- バックエンド: Spring Boot（Java 21）。既存の `POST /api/chat` の契約（`docs/api/chat-response-schema.md`）には変更を加えない。
- 応答検証は `chatApi.ts` と同じ方針（必須フィールド検証・未知フィールド無視・不正時はクラッシュせずエラー表示）に従う。
- モックデータはバックエンドで生成しフロントは描画に徹する、という本プロジェクトの既存方針に従う。

### 前提（調査により確認）

- 閾値は 警戒=70%以上、危険=90%以上 の2閾値・3段階とし、ノード（CPU/メモリの最悪値）とエッジ（帯域使用率）で共通とする。
- チャット画面への復帰導線は専用の「戻る」ボタンを設けず、サイドバーの既存操作（New Chat・会話選択・定型レポート・新規問い合わせ）で戻る方式とする。
- サイドバーの切替ボタンのラベルは「システムモニタリング」とする（設計時に変更可）。
- ランダムウォーク生成のため、バックエンドはこの機能に限りインメモリで前回値を保持する（チャット機能のステートレス方針の例外。サーバ再起動でリセットされてよい）。
- ノードの具体名・座標レイアウト・新エンドポイントのパス名/スキーマ詳細は設計フェーズで確定する。
- ポーリング間隔は5秒固定とし、設定変更UIは設けない。

## 重大な非機能要件

- 5秒間隔の再描画でブラウザが目に見えて重くならないこと（8ノード規模のSVG差し替えであれば通常問題にならない想定）。
- ライト/ダーク両テーマで正常/警戒/危険の3色が判別可能であること（色だけに依存せず数値・ゲージを併記するため色覚多様性にも一定配慮される）。

## 未決定事項

- スパイクの発生頻度・持続時間の具体値 — 選択肢: 頻度30秒〜2分に1回程度／持続10〜30秒 など — 推奨仮置き: 平均60秒に1回程度発生し15秒程度持続（デモで数分以内に必ず色変化が観察できる、という目的から逆算）／外れた場合の影響: デモの見栄えの問題にとどまり、生成ロジックの定数調整で対応可能。

## 決定記録

- Q1: モニタリングの配置場所 → 専用モニタリング画面（理由: 常時更新ダッシュボードはチャット履歴のUXに馴染まず、既存チャット応答スキーマ契約に手を入れずに済む）
- Q2: トポロジーとメトリクスのデータ源 → 構成もメトリクスもバックエンドの新APIが返す（理由: モックはバックエンドで生成しフロントは描画に徹する既存方針に一致）
- Q3: 更新の通信方式 → RESTポーリング5秒間隔（理由: 既存アーキテクチャがREST一本で、5秒間隔ならポーリングで十分・実装最小）
- Q4: 描画技術 → 自作SVG・固定座標レイアウト（理由: 既存グラフも自作でありライブラリ依存追加ゼロ。少ノードの固定構成なら十分）
- Q5: モック構成の規模 → 3層Web構成・8ノード程度（理由: ボトルネック表現のデモに十分な経路数があり、自作SVGで無理なく描ける規模）
- Q6: ノードの逼迫度表現 → 3段階色分け＋ミニゲージ＋数値、閾値70%/90%（理由: 一目で逼迫箇所が分かるUXと既存ゲージ表現の流用で実装が軽い）
- Q7: エッジのボトルネック表現 → 3段階色分け＋帯域使用率%ラベル（理由: ノードと同じ色ルールで統一され読みやすく、SVGのstroke色変更だけで実現可能）
- Q8: メトリクスの変動パターン → ランダムウォーク＋周期的スパイク（理由: 色変化・ボトルネック検知という機能の見せ場がデモで必ず確認できる）
- Q9: インタラクション → なし・表示専用（理由: ゲージ＋数値の常時表示で情報は足り、実装コスト最小）
- Q10: ポーリング失敗時の挙動 → 直前データ維持＋エラーバナー＋自動復帰（理由: 監視画面のUXとして自然で、クラッシュさせない既存エラーハンドリング方針と整合）
- Q11: ポーリングのライフサイクル → モニタリング画面表示中のみ（理由: 表示していない間の更新は誰も見ないため無駄な通信をしない最小実装）
- Q12: 図以外の画面要素 → 図＋最終更新時刻のみ（理由: 色分けされた図自体がサマリーの役割を果たすため追加要素は冗長）

## 用語集

- **逼迫具合**: 本機能ではノードのCPU使用率・メモリ使用率（0〜100%）を指す。閾値70%/90%で正常・警戒・危険に区分する。
- **ネットワークボトルネック**: ノード間接続（エッジ）の帯域使用率が警戒・危険域にある状態。エッジの色とラベルで表現する。
- **画面リフレッシュなしに再描画**: ページリロードを伴わないSPA内のリアクティブなDOM更新のこと（Vueのリアクティビティによる差分描画）。
- **スナップショット**: ある時点の全ノード・全エッジのメトリクス一式。本機能のAPIは履歴を持たず常に最新スナップショットのみを返す。
