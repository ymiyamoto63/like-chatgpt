# 問い合わせ送信前のFAQ提示機能

## 概要

既存の新規問い合わせウィザード（カテゴリ→緊急度→内容入力→確認→完了）に、FAQ提示ステップを挿入する。内容入力の直後（確認ステップの前）に、選択済みカテゴリに対応するFAQのタイトル3件をリンクとして提示する。リンクをクリックするとそのFAQの詳細がAI回答として表示され、あわせてFAQ一覧が再掲される。FAQで解決した場合はフローを終了し、解決しない場合は既存の確認ステップ（要約表→登録）に合流して実際の問い合わせを行う。既存方針どおりバックエンドのステートレスなMockが応答を駆動する。

## 背景・目的

問い合わせを登録する前に自己解決の機会を提供し、問い合わせ件数を抑制する「FAQディフレクション」のUXをチャットUI上でモック検証する。将来LLMに差し替えた際に「LLMが問い合わせ内容から関連FAQを提示し、解決しなければチケット登録に進む」構図をそのまま実現できるよう、FAQ一覧を応答スキーマのコンポーネントとして確立する。

## 対象外（Non-goals）

- 実際のLLM API連携、FAQデータの外部管理（DB・CMS・検索API）— FAQはバックエンドにハードコード
- 問い合わせ内容の語句によるFAQの絞り込み・検索（マッチングはカテゴリ別固定リストのみ）
- 通常チャット（ウィザード外）でのFAQ提示
- FAQ閲覧履歴・解決率などの記録・集計
- 既存ウィザードのFAQ以外のステップ（カテゴリ→緊急度→内容入力、確認→完了）の仕様変更
- 認証・永続化（既存方針踏襲）

## 機能要件

- FR-1: 内容入力ステップで累積文（カテゴリ/緊急度/内容）が送信されたとき、Mockは確認ステップの代わりにFAQ提示ステップを返す。応答は、案内テキスト（`reply`）＋ 選択済みカテゴリに対応するFAQタイトル3件のFAQリストコンポーネント ＋ 選択肢コンポーネント（「解決した」「解決しないので問い合わせる」）で構成する
- FR-2: チャット応答スキーマにFAQリストコンポーネント（type名は設計フェーズで決定。例 `faq_list`）を追加する。各項目はFAQタイトル（1件以上）。既存の `UiComponent` 判別共用体の拡張として実装し、契約文書に追記する
- FR-3: フロントエンドはFAQリストコンポーネントをアシスタント吹き出し内にリンク風テキスト（下線等、選択肢ボタンと視覚的に区別できる表現）の縦一覧として描画する。リンククリックで「FAQ: <タイトル>」等の定型文がユーザー発言として送信される
- FR-4: MockはFAQタイトルの定型文への応答として、FAQ詳細本文（`reply`）＋ 同カテゴリのFAQリストコンポーネント（一覧再掲）＋ 選択肢コンポーネント（「解決した」「解決しないので問い合わせる」）を返す。ユーザーは別のFAQを続けて閲覧できる
- FR-5: FAQデータは4カテゴリ（請求／技術／アカウント／その他）×各3件をバックエンドにハードコードする。「その他」には汎用FAQを用意する。FAQタイトルは全カテゴリを通じて一意とし、Mockはタイトルからカテゴリと本文を一意に解決できること（ステートレス維持のため）
- FR-6: 「解決した」を選ぶと、Mockは終了メッセージ（選択肢なし）を返してフローを終了する。以降その会話は通常のチャットとして継続入力できる
- FR-7: 「解決しないので問い合わせる」を選ぶと、フロントエンドが累積内容（カテゴリ/緊急度/内容）を含む定型文を組み立てて送信し、Mockは既存の確認ステップ（要約表＋「登録する」「やり直す」「キャンセル」）を返す。以降は既存フロー（登録→受付完了、やり直す→カテゴリ選択から再開、キャンセル→中断）のとおり
- FR-8: 入力欄無効化・過去メッセージ無効化の既存規則をFAQ提示にも適用する。FAQ提示ステップ・FAQ詳細応答は選択肢コンポーネントを含むため、最新発言である間はテキスト入力欄を無効化する。最新のアシスタント発言以外に含まれるFAQリンクは無効化して表示する（履歴として残す）
- FR-9: FAQ提示中に会話を切り替えて戻った場合やリロード後も、FAQリンクのクリックおよび「解決しないので問い合わせる」からのフロー継続ができること（累積内容は会話履歴中の直近の累積文から復元でき、フロントの一時状態のみに依存しない）
- FR-10: FAQフロー用の定型文（「FAQ: <タイトル>」「解決した」「解決しないので問い合わせる」、累積内容を含む定型文）が既存のレポートキーワード（「担当」「カテゴリ」「種別」「日別」）やアラートフローに誤マッチしないこと。既存の定型レポート・通常チャット・アラート・ウィザードの他ステップの挙動は変えない
- FR-11: FAQリストコンポーネントの検証（タイトル1件以上・文字列であること）を [chatApi.ts](../../frontend/src/api/chatApi.ts) に追加し、不正な場合は既存方針どおりエラー吹き出しにフォールバックする

## 画面・UIフロー

新規画面は追加しない。既存ウィザードの内容入力と確認の間にFAQステップが挿入される。

```
ユーザー: カテゴリ: 請求 / 緊急度: 高 / 内容: 請求額が二重になっている
ボット:   ご登録の前に、関連するFAQをご確認ください。
          ・請求額が二重に表示される場合           ← リンク風テキスト（FAQリスト）
          ・請求書の再発行方法
          ・支払い方法の変更手順
          [解決した] [解決しないので問い合わせる]    ← 選択肢ボタン。入力欄は無効
ユーザー: FAQ: 請求額が二重に表示される場合          ← リンククリックで送信
ボット:   （FAQ詳細本文）
          ・請求額が二重に表示される場合           ← 一覧再掲（別FAQも閲覧可）
          ・請求書の再発行方法
          ・支払い方法の変更手順
          [解決した] [解決しないので問い合わせる]
ユーザー: 解決しないので問い合わせる（＋累積内容を含む定型文）
ボット:   （要約表: カテゴリ/緊急度/内容）この内容で登録しますか？
          [登録する] [やり直す] [キャンセル]        ← 以降は既存フロー
```

- 「解決した」押下: 「解決してよかったです」等の終了メッセージで終了し、通常チャットに戻る
- FAQリンク・選択肢とも、押下できるのは常に最新のアシスタント発言内のもののみ（既存規則）

## 受け入れ基準

- AC-1: ウィザードでカテゴリ・緊急度を選び内容を入力すると、確認ステップではなく、選択したカテゴリのFAQタイトル3件のリンクと「解決した」「解決しないので問い合わせる」ボタンが表示される（FR-1, 3, 5。検証方法: 手動確認）
- AC-2: FAQリンクをクリックすると、そのFAQの詳細本文がAI回答として表示され、FAQ一覧と2つの選択肢が再掲される（FR-3, 4。検証方法: 手動確認）
- AC-3: FAQ詳細表示後、再掲された一覧から別のFAQを続けて閲覧できる（FR-4。検証方法: 手動確認）
- AC-4: 「解決した」を選ぶと終了メッセージが表示され、その会話で通常のチャットが継続できる（FR-6。検証方法: 手動確認）
- AC-5: 「解決しないので問い合わせる」を選ぶと入力済み内容の要約表つき確認ステップが表示され、「登録する」で受付番号つき完了メッセージに至る（FR-7。検証方法: 手動確認）
- AC-6: FAQ提示中はテキスト入力欄・送信ボタンが無効であり、過去の発言に含まれるFAQリンク・選択肢は無効化表示される（FR-8。検証方法: 手動確認）
- AC-7: `POST /api/chat` の統合テストで、(a) 累積文→FAQリスト＋選択肢を含む応答、(b) 「FAQ: <タイトル>」→詳細本文＋同カテゴリ一覧再掲＋選択肢、(c) 累積内容を含む未解決定型文→要約表＋確認選択肢、(d) 「解決した」→終了メッセージ、が契約どおり返る（FR-1, 2, 4, 6, 7。検証方法: 自動テスト（MockMvc））
- AC-8: 4カテゴリすべてについて、累積文に対しそのカテゴリに対応するFAQ 3件が返る（FR-5。検証方法: 自動テスト）
- AC-9: FAQフロー用定型文に対して既存のレポート応答・アラート応答が返らず、既存の定型レポート・通常チャット・ウィザード他ステップに退行がない（FR-10。検証方法: 既存＋追加の自動テスト、手動確認）
- AC-10: タイトル0件など不正なFAQリストコンポーネントを含む応答はエラー吹き出しにフォールバックし、アプリはクラッシュしない（FR-11。検証方法: 自動テストまたは手動確認）
- AC-11: [chat-response-schema.md](../chat-response-schema.md) にFAQリストコンポーネントの定義とサンプルが追記され、AC-7 のテストが検証する実応答と一致している（FR-2。検証方法: 手動確認）
- AC-12: FAQ提示中にリロードまたは会話切り替えを行って戻っても、FAQリンクのクリックと「解決しないので問い合わせる」による確認ステップへの合流ができる（FR-9。検証方法: 手動確認）

## 制約

- 技術スタックは既存踏襲: フロント Vue 3 + TypeScript + Pinia（Vite）、バックエンド Java Spring Boot + Maven
- バックエンドはステートレスを維持する。FAQステップの進行状態も「直前のユーザー発言の定型文」だけで判別できる設計とし、`ChatRequest`（`message` のみ）は変更しない。このため「解決しないので問い合わせる」の送信文には累積内容を含める（具体的な文言・書式は、Mockが一意に判別でき既存キーワードと衝突しないことを条件に設計フェーズで決定）
- FAQリストコンポーネントは既存の `UiComponent` 判別共用体（`@JsonTypeInfo`/`@JsonSubTypes`）の拡張として追加し、[chat-response-schema.md](../chat-response-schema.md) の「将来の拡張」手順に従う
- FAQタイトル・本文を含む応答スペック内の文字列は既存方針どおり `v-html` を用いずテキストとして描画する（XSS防止）。FAQリンクは `<a href>` による遷移ではなく、クリックで定型文を送信するインタラクティブ要素として実装する
- Mockの判定は問い合わせフロー系（FAQ含む）を既存レポートキーワードより先に評価する（既存方針踏襲）
- スタイルは既存のCSSカスタムプロパティとトーンに合わせ、「動くプロトタイプ」優先の既存方針を踏襲する

### 前提（調査・整合により確定）

- FAQ提示の初回（一覧のみの時点）にも「解決した」「解決しないので問い合わせる」の2択を併記する（詳細表示後と同一構成。タイトルを見ただけで解決に気づくケースを許容し、応答構成を1種類に保てる）
- FAQステップに独立した「キャンセル」選択肢は設けない（「解決した」がフロー終了、「解決しない」→確認ステップの「キャンセル」で中断でき、経路が重複するため）
- FAQ詳細は `reply` のテキストのみで表現する（表・グラフ等は用いない）
- 確認ステップの「やり直す」でカテゴリ選択から再開した場合、内容入力後に再びFAQ提示ステップを通る（フロー定義上の自然な帰結）
- 1つの応答に含まれるFAQリストコンポーネントは1個を想定する

## 重大な非機能要件

- **表示の安全性（XSS防止）**: FAQタイトル・本文は将来LLM由来になる前提で、HTMLとして解釈せず必ずテキストとして描画する（既存方針の踏襲）
- **不正応答への耐性**: 不正なFAQリストコンポーネントでフロントがクラッシュせずエラー表示にフォールバックする（FR-11。既存方針の踏襲）

## 決定記録

- Q1: FAQ提示のタイミング → 内容入力直後・確認ステップの前（理由: カテゴリ・内容が揃った状態で関連FAQを出せ、既存フローへの挿入点として自然）
- Q2: FAQの選定方法 → カテゴリ別の固定リスト（理由: 既存Mockのキーワードマッチ方針に合致し、デモで挙動が予測しやすい）
- Q3: FAQ詳細表示後の選択肢 → 「解決した」「解決しないので問い合わせる」＋FAQ一覧再掲（理由: 複数FAQを続けて閲覧でき行き止まりがない。既存の選択肢UIパターンで実現可能）
- Q4: 「解決した」時の挙動 → 終了メッセージでフロー終了（理由: 既存のキャンセル時と同じパターンで実装が最も単純）
- Q5: リンクのスキーマ表現 → 新規FAQリストコンポーネントを追加（理由: 「タイトルはリンク」という要件を見た目で満たしつつ選択肢ボタンと意味を分離。スキーマ拡張手順は契約文書に確立済み）
- Q6: リンククリック時の挙動 → 定型文を送信してバックエンド往復（理由: 既存の選択肢パターン踏襲でステートレス維持・履歴にも残り、「詳細がAI回答として提示される」という要件に合致）
- Q7: 未解決時の流れ → 累積内容を含む定型文を送信して既存の確認ステップに合流（理由: 登録前の最終確認が残り、既存フローをそのまま再利用できる）
- Q8: FAQ件数 → カテゴリごとに3件固定（理由: 一覧が一目で読める量で、モックデータの用意も現実的）
- Q9: カテゴリ「その他」の扱い → 全カテゴリで提示（汎用FAQを用意）（理由: スキップ分岐がなく実装・テストが単純で、デモの挙動が一貫する）

## 用語集

- **FAQ提示ステップ**: 内容入力後・確認前に挿入される、FAQタイトル一覧と解決/未解決の選択肢を提示する対話ステップ。実体はステートレスMockが累積文を判別して返す応答
- **FAQリストコンポーネント**: 応答JSONの `components` に新設する描画指示。FAQタイトルをリンク風テキストの一覧として描画し、クリックが定型文のユーザー発言送信になる（ボタン描画の選択肢コンポーネントとは別物）
- **解決／未解決**: 「解決した」＝問い合わせ登録を行わずフロー終了、「解決しないので問い合わせる」＝既存の確認→登録フローへ進むこと。データはどこにも保存されない（モック）
- **累積文**: 内容入力ステップでフロントが組み立てる「カテゴリ: X / 緊急度: Y / 内容: Z」形式の送信文（既存仕様）。本機能ではこれへの応答が確認ステップからFAQ提示ステップに変わる
