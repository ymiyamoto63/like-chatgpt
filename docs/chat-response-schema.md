# チャット応答 JSONスキーマ契約文書

> 本書は `docs/requirements.md`（FR-10, AC-8）および `docs/design.md`（「応答JSONスキーマ（確定仕様）」節）に基づき作成した、`POST /api/chat` の応答JSONの契約文書である。

## 位置づけ

- この応答JSONスキーマは、**将来LLMがそのまま出力する契約（フォーマット）**として設計されている。現時点ではLLMには接続しておらず、バックエンドのキーワードマッチによるモック実装（`MockChatService`）が本スキーマに従ったJSONを決定論的に生成して返している。
- 将来、モック実装をLLM呼び出しに置き換える際は、LLMの出力がこのスキーマに一致するように制御（プロンプト設計・出力パーサ等）すればよく、フロントエンドの描画層・検証層には変更が不要となることを意図している。
- 本書に記載するサンプルJSONは、`backend/src/test/java/com/example/chatbackend/ChatControllerTest.java` が検証する実際のレスポンス、および実際に起動したバックエンドへ `curl` で問い合わせた結果と一字一句一致している（AC-8）。

## トップレベル: `ChatResponse`

`POST /api/chat` は常にこの形の構造化JSONを返す。従来の `{"reply": string}` 形式（`EchoService` によるエコー応答）は廃止済み。

| フィールド | 型 | 必須 | 説明・制約 |
|---|---|---|---|
| `reply` | `string` | ○ | 応答の要約テキスト。UIコンポーネントの有無に関わらず常に存在し、非nullの文字列 |
| `components` | `UiComponent[]` | ○ | 0件以上のUIコンポーネントスペック配列。フィールド自体は常に存在し、`null` にはならない（該当コンポーネントがない場合は空配列 `[]`） |

```
ChatResponse
{
  "reply": string,
  "components": UiComponent[]
}
```

## `UiComponent`（判別共用体）

`components` 配列の各要素は `"type"` フィールドの値で種別を判別する共用体である。現時点でサポートするのは `table`・`bar_chart`・`choices` の3種類のみ。

`"type"` フィールドは各コンポーネントのレコード自体には定義されておらず、バックエンド側の Jackson 設定（`@JsonTypeInfo(use = Id.NAME, include = As.PROPERTY, property = "type")` + `@JsonSubTypes`）によりシリアライズ時に自動付与される。

### `TableComponent`（`"type": "table"`）

| フィールド | 型 | 必須 | 説明・制約 |
|---|---|---|---|
| `type` | `"table"` | ○ | 固定リテラル。自動付与される判別子 |
| `columns` | `string[]` | ○ | 列見出し。要素数は1以上 |
| `rows` | `string[][]` | ○ | 行データ。各要素（1行）の長さは必ず `columns.length` と一致する。セル値は表示用の文字列（数値も文字列化して格納する） |

```
TableComponent
{
  "type": "table",
  "columns": string[],
  "rows": string[][]
}
```

### `BarChartComponent`（`"type": "bar_chart"`）

| フィールド | 型 | 必須 | 説明・制約 |
|---|---|---|---|
| `type` | `"bar_chart"` | ○ | 固定リテラル。自動付与される判別子 |
| `title` | `string` | ○ | グラフ見出し |
| `labels` | `string[]` | ○ | 棒のラベル。要素数は1以上 |
| `values` | `number[]` | ○ | `labels` と同数。JSON number。バックエンド実装では `List<Double>` として保持しているため、シリアライズ結果は整数値でも `12.0` のように小数点付きで出力される（スキーマ上は整数・小数のどちらも許容） |

```
BarChartComponent
{
  "type": "bar_chart",
  "title": string,
  "labels": string[],
  "values": number[]
}
```

### `ChoicesComponent`（`"type": "choices"`）

アシスタント発言内に選択ボタン群として描画される。ユーザーがボタンをクリックすると、そのラベル文字列がそのままユーザー発言として `POST /api/chat` に送信される（表示ラベル＝送信文）。新規問い合わせフロー（後述）の各ステップで使用する。

| フィールド | 型 | 必須 | 説明・制約 |
|---|---|---|---|
| `type` | `"choices"` | ○ | 固定リテラル。自動付与される判別子 |
| `options` | `string[]` | ○ | 選択肢。要素数は1以上。各要素は表示ラベルであり、クリック時にそのまま送信される定型文でもある |

```
ChoicesComponent
{
  "type": "choices",
  "options": string[]
}
```

フロントエンドは、選択肢を押下できるのを「最新のアシスタント発言に含まれる `choices`」に限定し（過去の発言内の選択肢は無効化表示）、最新のアシスタント発言に `choices` が含まれる間はテキスト入力欄を無効化する。

## サンプルJSON（3シナリオ）

以下は実際にバックエンドを起動し、`ChatControllerTest` と同じ入力文を `POST /api/chat` に送信して得られたレスポンス（改行はドキュメント上の可読性のために追加。実際の1行JSONと内容は同一）。

### シナリオA: 担当者別（FR-4a）

トリガー: メッセージ文字列に「担当」を含む（例:「今月の問い合わせ件数を担当者別にまとめて」）。

```json
{
  "reply": "今月の担当者別の問い合わせ件数をまとめました。",
  "components": [
    {
      "type": "table",
      "columns": ["担当者", "件数"],
      "rows": [
        ["佐藤", "12"],
        ["鈴木", "9"],
        ["田中", "7"],
        ["高橋", "5"]
      ]
    },
    {
      "type": "bar_chart",
      "title": "担当者別 問い合わせ件数",
      "labels": ["佐藤", "鈴木", "田中", "高橋"],
      "values": [12.0, 9.0, 7.0, 5.0]
    }
  ]
}
```

### シナリオB: カテゴリ別（FR-4b）

トリガー: メッセージ文字列に「カテゴリ」または「種別」を含む（例:「カテゴリ別の問い合わせ件数を見せて」）。

```json
{
  "reply": "カテゴリ別の問い合わせ件数をまとめました。",
  "components": [
    {
      "type": "table",
      "columns": ["カテゴリ", "件数"],
      "rows": [
        ["請求", "15"],
        ["技術", "11"],
        ["アカウント", "6"],
        ["その他", "4"]
      ]
    },
    {
      "type": "bar_chart",
      "title": "カテゴリ別 問い合わせ件数",
      "labels": ["請求", "技術", "アカウント", "その他"],
      "values": [15.0, 11.0, 6.0, 4.0]
    }
  ]
}
```

### シナリオC: 非マッチ（FR-5）

トリガー: メッセージ文字列が「担当」「カテゴリ」「種別」のいずれも含まない（例:「こんにちは」）。

```json
{
  "reply": "『今月の問い合わせ件数を担当者別にまとめて』のように聞いてください。表やグラフでお答えします。",
  "components": []
}
```

判定順序: まず新規問い合わせフローの定型文（下記シナリオD群）に一致するか判定する（フロー定型文は「カテゴリ」等の既存キーワードを含むため最優先）。一致しなければ「担当」を含むか → シナリオA。含まなければ「カテゴリ」または「種別」を含むか → シナリオB。どちらも含まなければ → シナリオC。

## シナリオD: 新規問い合わせフロー

問い合わせチケット登録ウィザード。フロントエンドの「新規問い合わせ」ボタン押下でトリガー文が送信され、以降は `choices` の選択（＝定型文送信）でステップが進む。バックエンドはステートレスのため、各ステップは直前のユーザー発言の定型文のみで判別する。

| 受信メッセージ | 応答 |
|---|---|
| `新規問い合わせ` または `やり直す` | カテゴリ質問＋`choices`（`カテゴリ: 請求` / `カテゴリ: 技術` / `カテゴリ: アカウント` / `カテゴリ: その他` / `キャンセル`） |
| `カテゴリ: <値>`（累積形式でない） | 緊急度質問＋`choices`（`緊急度: 高` / `緊急度: 中` / `緊急度: 低` / `キャンセル`） |
| `緊急度: <値>` | `"問い合わせ内容を入力してください。"`（`components` 空。フロントは入力欄を有効化） |
| `カテゴリ: <値> / 緊急度: <値> / 内容: <本文>`（累積形式。内容入力時にフロントが組み立て） | 確認応答（下記サンプル） |
| `登録する` | `"問い合わせを受け付けました。受付番号: INQ-0001"`（`components` 空） |
| `キャンセル` | `"問い合わせの登録を中止しました。"`（`components` 空） |

### 開始応答（トリガー: `新規問い合わせ`）

```json
{
  "reply": "問い合わせのカテゴリを選んでください。",
  "components": [
    {
      "type": "choices",
      "options": ["カテゴリ: 請求", "カテゴリ: 技術", "カテゴリ: アカウント", "カテゴリ: その他", "キャンセル"]
    }
  ]
}
```

### 確認応答（トリガー: `カテゴリ: 請求 / 緊急度: 高 / 内容: 請求額が二重になっている`）

```json
{
  "reply": "以下の内容で登録してよろしいですか？",
  "components": [
    {
      "type": "table",
      "columns": ["項目", "内容"],
      "rows": [
        ["カテゴリ", "請求"],
        ["緊急度", "高"],
        ["内容", "請求額が二重になっている"]
      ]
    },
    {
      "type": "choices",
      "options": ["登録する", "やり直す", "キャンセル"]
    }
  ]
}
```

登録データはどこにも保存されない（受付完了メッセージを返すのみのモック）。

## フロントエンドの安全なフォールバック・将来拡張について

- フロントエンドは、応答JSONの `components` 配列内に**未知の `type` 値**や、必須フィールドの欠落・型不一致などの**不正な構造**が含まれていた場合でも、アプリ全体がクラッシュしないようにフォールバックする（FR-7 / AC-5）。検証で不正と判断した場合は、応答メッセージ全体を「解釈できない応答」として扱い、「応答を表示できませんでした」等のエラー文言の吹き出しを表示する（部分的に描画できるコンポーネントだけを表示する部分成功は行わない）。
- この検証は API クライアント層（`chatApi.ts`）で `reply` の型・`components` の配列性・各要素の `type` ごとの必須フィールド（`table` なら `columns`/`rows`、`bar_chart` なら `title`/`labels`/`values`、`choices` なら `options`（1件以上））を確認する形で行い、`type` がこの3値以外の場合や条件を満たさない要素がある場合は不正とみなす。
- **将来の拡張**（例: 折れ線グラフ・円グラフなど新しい `type` の追加）は、(1) バックエンド側で新しい `UiComponent` 実装を `@JsonSubTypes` に追加し、(2) フロントエンド側で対応する検証ロジックと描画コンポーネントを追加する、という形で行う想定である。新しい `type` が実装される前の段階でその `type` を含む応答を受け取った古いフロントエンドは、上記の「未知の `type` は不正」というルールにより安全にエラー表示へフォールバックし、クラッシュしない。
- 同様に、`ChatResponse` や各コンポーネントに将来LLMが未知の追加フィールドを含めて出力した場合も、フロントエンドは検証で使用するフィールドのみを見て判定するため、余分なフィールドの存在自体では不正と判定しない（無視する）想定である。
