# チャット応答への「ダッシュボード応答」追加

## 概要

チャット応答に、KPIカード列（`stat_cards`、前月比等の増減表示`delta`を含む）とドーナツチャート（`donut_chart`）という2つの新しいUIコンポーネントtypeを新設し、「今月のサマリーを見せて」等のキーワードに対して、これらに既存の`trend_chart`を加えた複数コンポーネントが1つの吹き出し内に「ミニダッシュボード」として並ぶ応答を返せるようにする。目的はデモの見栄え向上であり、`docs/api/chat-response-schema.md`の「将来の拡張」節に定められた新type追加の手順（バックエンドで`@JsonSubTypes`に追加、フロントエンドで検証ロジック＋描画コンポーネントを追加、未知typeは安全にエラーフォールバック）に厳密に従う。

## スコープ

- バックエンド: `stat_cards`用のUiComponent record（`StatCardsComponent`。前月比等の増減を示す`delta`フィールドを任意で持つ）と`donut_chart`用のUiComponent record（`DonutChartComponent`）を新設し、`UiComponent`の`@JsonSubTypes`に追加する
- バックエンド: `KeywordMatchReplyGenerationAdapter`にキーワード分岐（メッセージが「サマリー」または「ダッシュボード」を含む場合）を追加し、要約テキスト＋`stat_cards`＋`donut_chart`＋`trend_chart`（既存の日別問い合わせ件数シナリオのロジック・データをそのまま流用）から成る応答を返す
- フロントエンド: `types/chat.ts`に`StatCardsComponent`・`DonutChartComponent`の型定義を追加し、`UiComponentSpec`判別共用体に組み込む
- フロントエンド: `chatApi.ts`に両typeの検証関数（必須フィールドチェック。`delta`は任意フィールドとして検証）を追加する
- フロントエンド: 両typeの描画Vueコンポーネントを新規作成する（`StatCardsView.vue`はKPIカードのグリッド表示＋`delta`の符号に応じた矢印・色分け表示、`DonutChartView.vue`は自前SVGによるドーナツチャート描画。外部チャートライブラリは追加しない）
- フロントエンド: `MessageBubble.vue`に両コンポーネントの判別・描画分岐を追加する
- ドキュメント: `docs/api/chat-response-schema.md`に両typeのフィールド定義・サンプルJSON・判定順序への影響を追記する

## 非ゴール

- 実データ・実LLM連携（本機能もモック実装として、キーワードマッチ＋固定/準固定データで応答する。既存プロジェクト方針を踏襲）
- KPIカードやドーナツチャートのインタラクティブ機能（クリックでの絞り込み、ホバー詳細、アニメーション等）。静的な表示のみ（`delta`の矢印・色分けは静的な表示切り替えであり、インタラクティブ機能には該当しない）
- 折れ線グラフ等、`stat_cards`・`donut_chart`以外の新規type追加
- 既存5種（table / bar_chart / choices / trend_chart / faq_list）のスキーマ・挙動の変更
- 外部チャートライブラリの新規依存追加（ドーナツチャートは自前SVGで描画する）
- ダッシュボード応答をトリガーする自然文の網羅的なキーワード集合の設計（「サマリー」「ダッシュボード」の2語のsubstring一致のみとし、それ以外の言い回しへの対応は行わない）

## 機能要件

- FR-1: `UiComponent`の判別共用体に、`stat_cards`と`donut_chart`の2つの新typeを追加する。既存5種の`@JsonSubTypes`定義・シリアライズ結果は変更しない。
- FR-2: `StatCardsComponent`（`type: "stat_cards"`）は、KPIカードの配列`cards`（要素数1以上）を持つ。各カードは`label`（string、カード見出し）と`value`（string、表示値。既存`TableComponent`の`rows`と同様、数値も表示用文字列として格納する）を必須フィールドとして持ち、加えて`delta`（string、前月比等の増減を表す表示用文字列。例:`"+12%"`）を任意（省略可）フィールドとして持つ。
  - FR-2a: `delta`が指定されている場合、フロントエンドは文字列の先頭文字により表示を分岐する。先頭が`+`の場合は「増加」として緑色＋上矢印付きで表示し、先頭が`-`の場合は「減少」として赤色＋下矢印付きで表示する。それ以外（例:`"±0%"`・`"0%"`など、`+`/`-`のいずれでも始まらない文字列）は「増減なし」として矢印・色付けなしのプレーン文字列表示とする。
  - FR-2b: `delta`フィールドが省略されている場合（応答JSONにフィールドが存在しない、または値が`null`）、カードは`label`と`value`のみを表示し、矢印・色・delta文字列のいずれも表示しない（プレーン表示）。
- FR-3: `DonutChartComponent`（`type: "donut_chart"`）は、`title`（string）、`labels`（string[]、要素数1以上）、`values`（number[]、`labels`と同数）を持つ。既存`BarChartComponent`と同じ形状のフィールド構成とし、フロントエンドが`values`の合計に対する各要素の比率からドーナツの弧を計算する。
- FR-4: `KeywordMatchReplyGenerationAdapter`は、メッセージ文字列が「サマリー」または「ダッシュボード」を含む場合（新規問い合わせフロー・アラートフローのいずれにも一致しない場合に限る）、要約テキストと`[stat_cards, donut_chart, trend_chart]`から成る`components`配列を返す。`trend_chart`は既存の日別問い合わせ件数シナリオ（`dailyTrendScenario`）のタイトル・ロジック・データ（直近14日間、平均値の水平線付き）をそのまま流用する。
- FR-5: FR-4の判定は、既存の判定順序（新規問い合わせフロー → アラートフロー → 日別 → 担当 → カテゴリ/種別 → フォールバック）の中に割り込ませる。挿入位置は「カテゴリ/種別」判定の直後・フォールバックの直前とする（キーワード衝突なし。後述「判定順序・キーワード衝突の検討」参照）。トリガーキーワードは「サマリー」「ダッシュボード」の2語のsubstring一致のみとする（他の自然文パターンへの対応は非ゴール）。
- FR-6: フロントエンドの`chatApi.ts`検証ロジックに、`stat_cards`（`cards`が1件以上の配列で、各要素が`label`・`value`ともstringであること。`delta`は「フィールドが存在しない」または「string型の値を持つ」のいずれも有効とする任意フィールドとして扱う）と`donut_chart`（`title`がstring、`labels`が1件以上のstring配列、`values`が`labels`と同数のnumber配列）の必須フィールド検証を追加する。条件を満たさない場合・`type`が上記7値（既存5＋新規2）以外の場合は不正とみなし、既存方針どおり応答全体をエラー表示にフォールバックする（部分成功なし）。
- FR-7: フロントエンドに`StatCardsView.vue`（KPIカードをグリッド状に横並び表示）と`DonutChartView.vue`（自前SVGでドーナツ状の弧を`label`ごとに描画し、凡例として`label`と比率または`value`を表示）を新規作成し、`MessageBubble.vue`の描画分岐（`type`ごとのv-if/v-else-ifチェーン）に追加する。`StatCardsView.vue`は各カードに`label`・`value`を表示し、`delta`が指定されている場合はFR-2aの符号判定仕様に従って矢印記号（例:`▲`/`▼`）と色（緑/赤）を付与し、`delta`が省略されている場合（FR-2b）は矢印・色を付けない。既存の他コンポーネント（`TableView`等）と同様、`v-html`は使用せずテキスト・数値をそのままバインドする（XSS対策の既存方針を踏襲）。
- FR-8: `docs/api/chat-response-schema.md`に、`StatCardsComponent`（`delta`の任意性・符号判定仕様を含む）・`DonutChartComponent`のフィールド定義表・サンプルJSON・トリガーキーワードを追記し、判定順序表（シナリオA〜Fの記載パターンに倣う）にダッシュボードシナリオ（シナリオG想定）を追加する。

## 判定順序・キーワード衝突の検討

- 既存トリガーキーワードは「担当」「カテゴリ」「種別」「日別」、および新規問い合わせフロー・アラートフローの各定型文（「新規問い合わせ」「カテゴリ: 」「緊急度: 」「FAQ: 」「解決した」「解決しないので問い合わせる」「登録する」「やり直す」「キャンセル」「使用率が高い原因を教えて」「CPU」「メモリ」「トラフィック」「閉じる」）である。新規トリガー「サマリー」「ダッシュボード」（この2語のsubstring一致のみで確定）は、いずれの既存キーワード文字列とも部分一致・包含関係がなく、衝突しないことをリポジトリ全文grepで確認済み。
- 例文「今月のサマリーを見せて」は「担当」「カテゴリ」「種別」「日別」のいずれも含まないため、既存分岐（FR-4a/4b/日別）より先に判定しても後に判定しても結果は変わらない。本要件では既存分岐を変更しない設計方針（非ゴール）を優先し、既存5分岐の最後（カテゴリ/種別判定の直後）に新規分岐を追記する形とする。これにより既存の`generateReply`の早期リターン群への差分が最小になる。
- 将来「サマリー」「ダッシュボード」を含む定型文が新規問い合わせフローやアラートフローに追加された場合は再度の衝突確認が必要になる点を、実装コメントとして残すことを推奨する（設計フェーズで判断）。

## 制約

- 既存の応答JSONスキーマ拡張手順（`docs/api/chat-response-schema.md`「将来の拡張」節）に従うこと: (1)バックエンドで`@JsonSubTypes`に追加、(2)フロントエンドで検証ロジック＋描画コンポーネントを追加、(3)未知typeは安全にエラーフォールバック（部分成功はしない）という既存方針を変更しない。
- ドーナツチャートは外部チャートライブラリを追加せず、自前SVGで描画する（タスク指定の制約）。
- バックエンドはステートレスを維持し、`ApplicationConfiguration`のBean配線・`KeywordMatchReplyGenerationAdapter`のヘキサゴナル構成上の位置づけ（`ReplyGenerationPort`の実装アダプタ）を変更しない。
- `stat_cards`のカード値はテーブルコンポーネントと同様に表示用文字列として保持する（数値演算はバックエンド側で完結させ、フロントは表示に徹する。既存`TableComponent`の設計方針を踏襲）。
- `StatCardsComponent`の`delta`フィールドは省略可能（`null`許容）とする。既存の`UiComponent`実装record群は全フィールドが必須であり、任意フィールドを持つのは本コンポーネントが初めてとなる。`null`値をJSONでどう表現するか（フィールド自体を省略する`@JsonInclude(NON_NULL)`等の設定か、明示的に`null`を出力するか）は設計フェーズで決定してよいが、フロントエンドの検証（FR-6）はいずれの表現も受理できるようにする。
- フロントエンドに自動テストランナーは未導入のため、フロント側の検証は`npx vue-tsc -b`・`npm run build`のグリーン維持と手動確認で行う（`pipeline-config.md`の`testing`節を踏襲）。
- モックの具体的なKPI値・カテゴリ内訳の数値は設計・実装フェーズで自由に決めてよい（既存`dynamic-ui-generation.md`の前提を踏襲。整合性のある架空データであれば足りる）。

## 用語集

- **ミニダッシュボード応答**: 1回のチャット応答の中に、KPIカード列（`stat_cards`）・ドーナツチャート（`donut_chart`）・トレンドチャート（`trend_chart`、既存流用）の3コンポーネントが並んで表示される応答のこと。専用画面を新設するものではなく、既存のチャット吹き出し内での複数コンポーネント併記（既存`components`配列の仕組み）をそのまま用いる。
- **delta**: `StatCardsComponent`の各カードが任意で持つ、前月比等の増減を表す表示用文字列（例:`"+12%"`）。先頭文字が`+`なら増加（緑/上矢印）、`-`なら減少（赤/下矢印）、それ以外は増減なし（矢印・色なしのプレーン表示）として描画する。フィールド自体が省略されている場合もプレーン表示（矢印・色なし）となる。
