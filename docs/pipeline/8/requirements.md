# サイドバーへの「ダッシュボード」ボタン追加

## 概要

左サイドバーの「新規問い合わせ」ボタンと「システムモニタリング」ボタンの間に、「ダッシュボード」ボタンを新設する。押下すると、直前のパイプライン（Issue #6 / PR #7、マージ済み）で実装済みのミニダッシュボード応答（シナリオG: KPIカード`stat_cards`＋ドーナツチャート`donut_chart`＋トレンドチャート`trend_chart`）を、既存の「新規問い合わせ」「定型レポート」ボタンと同じ操作パターン（新規会話を作成し、トリガー文言をユーザー発言として送信）でワンクリック呼び出せるようにする。バックエンドの応答生成ロジックは既に実装済みのため、本タスクはフロントエンドのみで完結する見込み。

## スコープ

- `frontend/src/components/layout/Sidebar.vue`: 「新規問い合わせ」ボタンと「システムモニタリング」ボタンの間に「ダッシュボード」ボタンを追加する
- ボタン押下時、既存の類似ボタン（`handleNewInquiry`／`handleReport`）と同じ組み合わせ（`monitoringStore.showChat()` → 新規会話作成 → シナリオGトリガーに一致するメッセージを送信）で動作するハンドラを実装する
- ボタンの活性・非活性（`isLoading`連動）およびスタイル（既存ボタン群のトーンとの整合）を、既存ボタンとの一貫性を根拠に決定する（詳細は機能要件を参照）

## 影響範囲

- **フロントエンド 画面コンポーネント** — `frontend/src/components/layout/Sidebar.vue`（ボタン追加とクリックハンドラ）
- **フロントエンド 状態管理** — `frontend/src/stores/conversationStore.ts`: 既存の`sendReportPrompt`/`startInquiry`と同型の呼び出し（`createConversation()` + `sendMessage()`）で実現できる見込み。新規ストアアクションを追加するかは設計フェーズの判断
- 変更しない領域（調査により確認済み。詳細は制約を参照）:
  - バックエンド全般（`KeywordMatchReplyGenerationAdapter`のシナリオGトリガー「サマリー」「ダッシュボード」は実装済み。追加のキーワードやAPI変更は不要）
  - `frontend/src/types/chat.ts`・`frontend/src/api/chatApi.ts`（`stat_cards`/`donut_chart`/`trend_chart`の型・検証は実装済み）
  - `frontend/src/constants/reportButtons.ts`およびバックエンドの`CannedPhraseSuggestionAdapter`（フロント⇔バック同期定数。本ボタンは「定型レポート」欄ではなく上部の独立ボタンであり、メッセージ入力欄の入力補完候補に載せる要件ではないため対象外）
  - `docs/api/chat-response-schema.md`（応答スキーマ自体は変更しない。呼び出し口が増えるだけ）

## 非ゴール

- バックエンドの変更（トリガーキーワード追加、応答内容の変更等）
- `frontend/src/constants/reportButtons.ts` / `CannedPhraseSuggestionAdapter`（入力補完の定型文リスト）への「ダッシュボード」追加
- ダッシュボード応答の内容（KPI値・カテゴリ内訳・トレンドデータ）自体の変更
- 専用の「ダッシュボード画面」の新設（あくまで既存のチャット吹き出し内複数コンポーネント表示を呼び出す入口であり、`activeScreen`に新しい状態を追加するものではない）
- 「新規問い合わせ」ボタンや「定型レポート」ボタン群、既存の会話履歴・リネーム・削除機能の挙動変更

## 機能要件

- FR-1: サイドバーの「新規問い合わせ」ボタンと「システムモニタリング」ボタンの間に、ラベル「ダッシュボード」のボタンを追加する。
- FR-2: ボタン押下時、`monitoringStore.showChat()` を呼び出してチャット画面に切り替えたうえで、常に新規会話を作成・アクティブ化し、固定文言「ダッシュボードを表示して」をユーザー発言として送信する（既存会話には送信しない）。この文言は `docs/api/chat-response-schema.md` シナリオGのトリガー条件（メッセージ文字列に「サマリー」または「ダッシュボード」を含む）に一致する。常に新規会話を作成する挙動は、隣接する「新規問い合わせ」ボタン（`handleNewInquiry`）・「定型レポート」ボタン群（`handleReport`）と同一パターンであり、進行中のチャットを汚さない既存方針（`docs/requirements/sidebar-report-buttons.md`決定記録Q2・`docs/requirements/new-inquiry-wizard.md`決定記録Q4）に揃える。
- FR-3: 応答待ち中（`store.isLoading`）はボタンを無効化する。根拠: 既存のメッセージ送信系ボタン（「新規問い合わせ」「定型レポート」各ボタン）は例外なく`:disabled="store.isLoading"`を持ち、多重送信で会話が複数作られることを防いでいる。画面切替のみを行う「システムモニタリング」ボタン（メッセージを送信しない）だけがこの制御を持たない。ダッシュボードボタンはメッセージを送信するため前者のパターンに従う。
- FR-4: ボタンの視覚表現は「新規問い合わせ」ボタンと同系のセカンダリボタンスタイルとし、`monitoringStore.activeScreen`に連動したハイライト（「システムモニタリング」ボタンが持つアクティブ表示）は付与しない。ダッシュボードは専用画面ではなく、押した瞬間にメッセージを1件送信するだけの単発アクション（「定型レポート」ボタン群や「新規問い合わせ」ボタンと同種）であるため、押しっぱなしを表すハイライト状態を持つ概念がそもそも存在しない。
- FR-5: 本変更はフロントエンドのみで完結し、バックエンド（`KeywordMatchReplyGenerationAdapter`等）・型定義・APIクライアント・レスポンススキーマドキュメントには変更を加えない。

## 受け入れ基準

- AC-1: サイドバーに「新規問い合わせ」「ダッシュボード」「システムモニタリング」の順でボタンが並んで表示される（FR-1。検証方法: 手動確認）
- AC-2: 「ダッシュボード」ボタンを押下すると、新規会話が作成・アクティブ化され、ユーザー吹き出しに送信文言、続けてアシスタント吹き出しに`stat_cards`（KPIカード）・`donut_chart`（ドーナツチャート）・`trend_chart`（トレンドチャート）の3コンポーネントを含むミニダッシュボード応答が表示される（FR-2。検証方法: 手動確認）
- AC-3: 進行中の会話が選択された状態で「ダッシュボード」ボタンを押しても、その会話にはメッセージが追加されず、別の新規会話に送信される（FR-2。検証方法: 手動確認。自動テストランナー未導入のため）
- AC-4: モニタリング画面（`activeScreen === 'monitoring'`）表示中に「ダッシュボード」ボタンを押すと、チャット画面に切り替わったうえでAC-2の応答が表示される（FR-2。検証方法: 手動確認）
- AC-5: 応答待ち中（`isLoading`）は「ダッシュボード」ボタンが無効化され、連打しても会話が1つしか作られない（FR-3。検証方法: 手動確認）
- AC-6: 「ダッシュボード」ボタンには`activeScreen`連動のハイライト（色反転等）が付かず、常に「新規問い合わせ」ボタンと同じ見た目で表示される（FR-4。検証方法: 手動確認）
- AC-7: `cd frontend && npx vue-tsc -b` および `cd frontend && npm run build` が成功する（FR-1〜4の変更後も型エラーがない。検証方法: 自動テスト（コマンド実行）。`vue-tsc --noEmit`は本リポジトリでは no-op のため使用しない）
- AC-8: 既存の「新規問い合わせ」「定型レポート」「システムモニタリング」ボタン、会話履歴の選択・リネーム・削除の挙動が本変更前後で変わらない（FR-1の非干渉確認。検証方法: 手動確認）

## 制約

- 技術スタックは既存踏襲: フロント Vue 3 + TypeScript + Vite + Pinia + Tailwind CSS 4。フロントエンドに自動テストランナーは未導入のため、検証は`npx vue-tsc -b`・`npm run build`のグリーン維持と手動確認で行う。
- バックエンドはヘキサゴナル構成・ステートレスを維持しており、シナリオGのトリガー判定（`KeywordMatchReplyGenerationAdapter`、メッセージに「サマリー」または「ダッシュボード」を含む場合に応答）は実装済みのため、本タスクでは一切変更しない。
- 送信は既存の`conversationStore`のアクション経由とし、`chatApi.ts`・応答スキーマには手を入れない。
- フロント⇔バック同期定数（`reportButtons.ts` ⇔ `CannedPhraseSuggestionAdapter`）は本タスクの対象外（非ゴール参照）。触れる場合は必ず両側同時変更が必要という既存制約自体は変わらないが、今回は触れない前提とする。
- `.vscode/settings.json` はユーザー個人設定のためコミット・変更禁止。

## 決定記録

- Q1: 送信メッセージ文言 → 「ダッシュボードを表示して」で確定（理由: シナリオGトリガー「サマリー」「ダッシュボード」のsubstring一致に合致し、自然な操作指示文。会話タイトル規則にもそのまま収まる）
- Q2: クリック時の挙動 → 常に新規会話を作成して送信、で確定（理由: 隣接する「新規問い合わせ」・「定型レポート」ボタンと同じパターンで、進行中のチャットを汚さない既存方針に整合）
- Q3: ボタンの見た目 → ハイライトなし。「新規問い合わせ」ボタンと同じセカンダリボタン様式で確定（理由: ダッシュボードは専用画面ではなく単発のメッセージ送信アクションであり、`activeScreen`連動ハイライトの対象になり得る「画面」概念に該当しない）
