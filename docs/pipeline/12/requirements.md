# ノード別スパークライン＋履歴チャート

## 概要

システム構成図モニタリング画面（[TopologyDiagram.vue](../../frontend/src/components/monitoring/TopologyDiagram.vue)）の各ノードに、CPU・メモリ使用率の直近の推移を示すスパークライン（小型折れ線）を埋め込み表示する。既存のCPU/メモリのゲージバー（帯グラフ）表示はスパークラインに置き換える。さらに、ノードをクリックすると、そのノードのCPU・メモリ使用率の推移をモーダルダイアログ上のチャートで表示する。履歴データはバックエンドを変更せず、フロントエンドがポーリングで取得したスナップショットをPiniaストア（[monitoringStore.ts](../../frontend/src/stores/monitoringStore.ts)）内にリングバッファとして蓄積することで実現する。

## 背景・目的

既存の`system-monitoring-view.md`では「メトリクス履歴の保存・時系列グラフ表示」「ノードのクリック等のインタラクション（詳細パネル、ツールチップ含む）」を明示的にNon-goalsとして除外していた。本機能はこの2点を撤回し、現在値のスナップショット表示に留まっていたモニタリング画面に「推移が見える」ダッシュボードらしい表現力を追加する。実監視システムとの接続は行わず、既存のモックメトリクス生成（ランダムウォーク＋周期的スパイク）をそのまま活用し、表現力の検証を行うことが狙い。

## スコープ

- フロントエンド（`monitoringStore`）で、ポーリングごとに取得したスナップショットから、ノードごとのCPU使用率・メモリ使用率の系列を対象ごとに直近36件（5秒間隔ポーリングで3分相当）を上限とするリングバッファとして蓄積する
- 各ノードのCPU・メモリの既存ゲージバー（帯グラフ）表示を、蓄積した履歴を用いたスパークライン（小型折れ線）表示に置き換える。ノードの現在値（%数値）表示、CPU・メモリ最悪値によるノード枠の3段階色分けは維持する
- ノードクリックで、そのノードのCPU・メモリ使用率の推移をモーダルダイアログとして画面中央にオーバーレイ表示する新規コンポーネントを追加する
- モーダル内のチャートは、CPU・メモリを1つのグラフに2本の折れ線として重ねて表示する（Y軸0〜100%固定）
- モーダル表示中もモニタリング画面のポーリングを継続し、チャートをリアルタイムに更新する
- 蓄積済みの履歴データは、チャット画面への切り替え（ポーリング停止）をまたいでも破棄せず、モニタリング画面へ戻った際に引き続き利用する

## 対象外（Non-goals）

- エッジ（ノード間接続）のスパークライン・履歴チャート表示（本機能はノードのCPU・メモリ使用率のみを対象とする。エッジの表示は既存の帯域使用率ラベル＋3段階色分けのまま変更しない）
- 履歴データのバックエンドでの保持、および履歴取得専用の新規APIエンドポイントの追加（履歴はフロントエンドのみで蓄積する）
- 履歴データの永続化（localStorage等への保存）およびブラウザのリロード・タブクローズ後の履歴復元
- 既存のアラート機能（[monitoringAlert.ts](../../frontend/src/constants/monitoringAlert.ts) / アラートバナー）との連携（例: アラートバナークリックで該当ノードの履歴チャートを開く導線）
- チャートのズーム・期間選択・表示件数変更などの追加インタラクション
- モーダル以外の詳細表示手段（サイドパネル常駐化、SVG内インライン展開）
- 履歴データのCSV等へのエクスポート

## 影響範囲

- **フロントエンド 画面コンポーネント**: [TopologyDiagram.vue](../../frontend/src/components/monitoring/TopologyDiagram.vue)（ゲージバー描画をスパークライン描画に置き換え、ノードのクリックハンドラを追加）、新規モーダルコンポーネント（例: ノード履歴チャートモーダル）、[MonitoringView.vue](../../frontend/src/components/monitoring/MonitoringView.vue)（モーダルの表示制御の組み込み）
- **フロントエンド 状態管理**: [monitoringStore.ts](../../frontend/src/stores/monitoringStore.ts)にノードごとのCPU・メモリ履歴（リングバッファ）を保持・更新するロジックと、選択中ノード（モーダル表示対象）の状態を追加。既存の`activeScreen`/`snapshot`/`polling`/アラート関連ロジックへの影響は最小限にする
- **フロントエンド 型定義**: [types/monitoring.ts](../../frontend/src/types/monitoring.ts)または新規ファイルに、履歴サンプル・履歴系列を表す型を追加
- **フロントエンド 定数**: [constants/monitoring.ts](../../frontend/src/constants/monitoring.ts)または新規ファイルに、履歴保持件数（36件）などの定数を追加
- **バックエンド**: 変更なし（`MonitoringController`/`MonitoringService`/`RandomWalkMetricsGenerationAdapter`/応答スキーマはすべて無変更）
- **ドキュメント**: 本ドキュメントで完結させ、[system-monitoring-view.md](system-monitoring-view.md)自体の書き換えは行わない（Non-goalsの一部撤回は本書の背景・目的に記載済み）

## 機能要件

- FR-1: フロントエンドは、モニタリング画面表示中にポーリングで取得した各スナップショットについて、ノードごとのCPU使用率・メモリ使用率をタイムスタンプ付きの時系列データとして`monitoringStore`内に蓄積する。
- FR-2: 蓄積する履歴は、対象（各ノードのCPU系列・メモリ系列）ごとに直近36件を上限とするリングバッファとし、上限を超えたら最も古いサンプルから破棄する。
- FR-3: 各ノードは、既存のCPU・メモリのゲージバー（帯グラフ）表示に代えて、CPU用・メモリ用それぞれのスパークライン（小型折れ線）を表示する。ノードの矩形サイズ・SVG上の座標（`monitoringLayout.ts`の`NODE_WIDTH`/`NODE_HEIGHT`/`NODE_POSITIONS`）は変更しない。現在値の数値（%）表示と、CPU・メモリ最悪値によるノード枠の3段階色分け（既存仕様）は維持する。
- FR-4: スパークラインは、そのノードについて蓄積済みの履歴（最大36件）を用いて描画する。画面表示直後などで履歴が1件しかない場合でも、エラーにせず点または短い線として描画する。
- FR-5: ノード（SVG上の矩形要素）をクリックすると、そのノードのCPU使用率・メモリ使用率の履歴チャートをモーダルダイアログとして画面中央にオーバーレイ表示する。
- FR-6: モーダルの履歴チャートは、選択ノードのCPU使用率とメモリ使用率を1つのグラフ上に2本の折れ線として重ねて表示する。Y軸は0〜100%固定とし、凡例でCPU/メモリを判別できるようにする。X軸（時間軸）は蓄積済み履歴データの範囲を表示する。
- FR-7: モーダル表示中もモニタリング画面のポーリングは継続し、新しいサンプルが到着するたびにモーダル内のチャートもリアルタイムに再描画される。
- FR-8: モーダルは閉じる操作（設計フェーズで具体化。例: 閉じるボタン・背景クリック・Escキー等のうち少なくとも1つ）で閉じ、モニタリング画面の通常表示に戻る。
- FR-9: 蓄積済みの履歴データは、モニタリング画面からチャット画面への切り替え（ポーリング停止）によって破棄しない。再度モニタリング画面を表示すると、蓄積済みの履歴を引き継いでスパークライン・モーダルチャートに反映する（ポーリング停止中は新規サンプルが追加されないだけで、既存データは保持される）。ブラウザのリロードやタブクローズ後は履歴が失われてよい（既存のデータ非永続化方針を踏襲）。
- FR-10: 本機能はエッジ（ノード間接続）のスパークライン・履歴チャートを提供しない。エッジの表示は既存の帯域使用率ラベル＋3段階色分けのまま変更しない。

## 画面・UIフロー

- モニタリング画面の既存構成（タイトル・最終更新時刻・アラートバナー・エラーバナー・構成図SVG）に、条件描画のモーダルを追加する。それ以外の既存レイアウトは変更しない。
- ユーザーが構成図上のいずれかのノードをクリックする → モーダルダイアログが開き、そのノードのCPU/メモリ履歴チャート（2本の折れ線を重ねたグラフ）が表示される。
- モーダル表示中もポーリングは継続し、5秒ごとにチャートへ新しいサンプルが追加されて更新される。
- モーダルを閉じる操作を行う → モーダルが閉じ、通常のモニタリング画面表示に戻る。構成図・ポーリングの状態は変わらない。
- サイドバーの操作でチャット画面に切り替える → モーダルが開いていれば閉じ、既存のチャット画面遷移が行われる。蓄積済みの履歴データは`monitoringStore`内に保持されたまま維持される。
- 既存のダークモード対応（Tailwindの`dark:`バリアント）に追随し、ライト/ダーク両テーマでスパークライン・モーダルチャートが判読できること。

## 受け入れ基準

- AC-1: モニタリング画面を数十秒以上表示していると、各ノードのCPU・メモリ表示が既存のゲージバーからスパークライン（小型折れ線）に置き換わっており、直近の推移が視覚的に確認できる（FR-3, FR-4）。（検証方法: 手動確認）
- AC-2: いずれかのノードをクリックすると、モーダルダイアログが開き、そのノードのCPU使用率・メモリ使用率が2本の折れ線として1つのチャートに重ねて表示される（FR-5, FR-6）。（検証方法: 手動確認）
- AC-3: モーダルを表示したまま5秒以上待つと、チャートに新しいサンプルが追加されて更新される（FR-7）。（検証方法: 手動確認）
- AC-4: モーダルを閉じる操作を行うと、モーダルが閉じてモニタリング画面の通常表示に戻る（FR-8）。（検証方法: 手動確認）
- AC-5: モニタリング画面を表示した直後（履歴が1件しかない状態）でも、スパークラインやモーダルチャートがエラーにならず正常に描画される（FR-4）。（検証方法: 手動確認）
- AC-6: モニタリング画面からチャット画面に切り替えて再度モニタリング画面に戻ると、切り替え前に蓄積されていた履歴がスパークラインに反映されている（FR-9）。（検証方法: 手動確認）
- AC-7: モニタリング画面を3分以上表示し続けても、履歴データの上限（36件）を超えて際限なくメモリが増え続けない（古いサンプルが破棄される）（FR-2）。（検証方法: 手動確認。確認を早めるため、上限件数を一時的に小さい値に変更して動作確認してよい）
- AC-8: 本機能追加後も、既存のモニタリング機能（トポロジー表示・3段階色分け・ポーリング・エラーバナー・アラートバナー）が退行しない（検証方法: 手動確認）

## 制約

- 技術スタックは既存踏襲で変更不可: フロントエンド Vue 3（Composition API）＋ Pinia ＋ Tailwind CSS 4。バックエンド Spring Boot（Java 21）は本機能では変更しない。
- 描画ライブラリ・チャートライブラリは導入しない（既存の構成図SVG・棒グラフ・トレンドグラフも自作であることに合わせ、スパークライン・モーダルチャートも自作SVGで実装する）。
- 履歴データの保持・蓄積はフロントエンド（`monitoringStore`）の責務とし、バックエンドに履歴保持ロジックを追加しない（バックエンドは引き続きステートレス、既存の`RandomWalkMetricsGenerationAdapter`の前回値保持のみを例外とする既存方針を維持）。
- ノードのSVG座標・サイズ（`monitoringLayout.ts`）は変更しない。
- 応答検証・エラーハンドリングは既存方針（`monitoringApi.ts`、ポーリング失敗時のエラーバナー等）をそのまま維持し、本機能による変更は行わない。
- 過度な作り込みをせず「動くプロトタイプ」を優先する既存方針を踏襲する。

## 重大な非機能要件

なし（履歴保持件数はノード9件×CPU/メモリ2系列×36件＝648サンプル程度であり、性能・メモリへの影響は軽微と判断）

## 未決定事項

なし（インタビューで主要な曖昧点はすべて解消済み）

## 決定記録

- Q1: 履歴データの保持元・蓄積方法 → フロントエンドで蓄積（`monitoringStore`内のリングバッファ）（理由: 既存の「モニタリング画面表示中のみ通信する」方針・バックエンドのステートレス方針を維持でき、実装コストが最小のため）
- Q2: 履歴として保持する期間 → 3分相当（36ティック、5秒間隔ポーリング前提）（理由: スパークラインで直近の傾向が見えつつ、モーダルの詳細チャートでも十分な変動が確認できる長さであり、メモリ・描画コストも小さいため）
- Q3: ノード内のスパークラインと既存ゲージバーの共存方法 → ゲージバーをスパークラインに置き換える（理由: ノードの枠サイズ・固定座標レイアウトを変更せずに実装でき、既存の数値表示・3段階色分けの枠色はそのまま維持できるため）
- Q4: ノードクリックで表示する履歴チャートの表示形式 → モーダルダイアログ（理由: 既存のMonitoringView.vueの縦積みレイアウトや固定座標のSVG構成に影響を与えず追加でき、実装コストが最小のため）
- Q5: モーダルの履歴チャートにおけるCPU・メモリの表示方法 → 1つのチャートに2本の折れ線を重ねて表示（理由: CPUとメモリを同時に比較でき、タブ切り替えのような追加UIの実装が不要なため）
- Q6: 画面遷移（モニタリング→チャット→モニタリング）時の蓄積済み履歴の扱い → 保持され、再訪問時に続きから表示する（理由: Piniaストアの状態を明示的にクリアしない実装で実現でき、再訪問直後からすぐに意味のあるグラフが見えるため）

## 用語集

- **スパークライン**: 本機能では、ノード内に埋め込む軸ラベルなしの小型折れ線グラフを指す。CPU用・メモリ用をそれぞれ独立して表示する。
- **履歴チャート**: 本機能では、ノードクリックでモーダルに表示する、CPU・メモリ使用率を2本の折れ線として重ねた通常サイズのグラフを指す。スパークラインと同じ蓄積データ（最大36件）を描画に用いる。
- **リングバッファ**: 対象（ノードのCPU系列・メモリ系列）ごとに、直近36件を上限として古いサンプルから自動的に破棄しながら保持するデータ構造を指す。
