# ストリーミング風タイプライター応答＋コンポーネントのフェードイン演出

## 概要・背景

現在、アシスタントの応答（`MessageBubble.vue`）はテキストとUIコンポーネント（表・棒グラフ・トレンドグラフ・ドーナツグラフ・統計カード・選択肢・FAQ一覧）を、応答受信と同時にすべて一括表示している（`POST /api/chat` はテキスト＋コンポーネント配列を1回のJSONで返す非ストリーミングAPIであり、バックエンド側にストリーミング配信の仕組みはない）。ChatGPT風の「じわじわ生成されている」体感を演出するため、(1) 応答テキストを1文字ずつ時間差で表示するタイプライター風アニメーション、(2) テキスト表示完了後に表・グラフ等のコンポーネントをフェードインで出現させる演出、をフロントエンドの表示層に追加する。バックエンドAPI・応答スキーマは変更しない、見た目のみの改善。

## スコープ

- アシスタントの新規応答メッセージについて、`message.content` を1文字ずつ時間差で描画するタイプライター表示を実装する。
- タイプライター表示完了後、当該メッセージに含まれるUIコンポーネント（`message.components`）を全て同時にフェードインで表示する。
- 演出中は該当メッセージの選択肢／FAQボタンをクリック不可、`MessageInput` を送信不可のままにする。
- 演出は「今回の画面表示中に新規到着したメッセージ」にのみ適用し、会話履歴の復元（会話切替・再読み込み）では即時に全体表示する。
- モニタリングアラート起点で直接ストアに追加されるアシスタントメッセージ（`createAlertConversation`）にも同様に適用する。
- 通信エラー・応答形式不正時のエラーメッセージ吹き出しにも同じ演出を適用する（別扱いにしない）。

## 対象外（Non-goals）

- バックエンドの実ストリーミング配信（SSE・WebSocket・chunked transfer 等）の導入。`POST /api/chat` の応答形式・通信方式は変更しない（`dynamic-ui-generation.md` の non-goals でも「ストリーミング応答は対象外」と既定済み。今回の「ストリーミング風」はフロントのみの疑似演出を指す）。
- 文字送り速度のユーザー設定・カスタマイズ機能。
- アニメーションのスキップ操作（クリック等で全文を即時表示させるUI操作）。
- `prefers-reduced-motion` 等のアクセシビリティ設定への個別対応。
- 複数コンポーネントの個別スタガー（時間差）フェードイン。全コンポーネント同時フェードインで確定（決定記録 D-3 参照）。
- 表・グラフ以外の新規UIコンポーネント種別の追加、既存コンポーネントの表示内容・データ仕様の変更。
- 会話タイトル生成ロジック、応答内容（キーワードマッチ等）の変更。

## 影響範囲

- **フロントエンド 画面コンポーネント**: `frontend/src/components/chat/MessageBubble.vue`（テキストのタイプライター表示、コンポーネントのフェードイン制御の追加）
- **フロントエンド 画面コンポーネント**: `frontend/src/components/chat/ChatWindow.vue`（`choices-enabled` 判定に演出完了状態を反映する必要がある可能性）
- **フロントエンド 画面コンポーネント**: `frontend/src/components/chat/MessageInput.vue` の呼び出し元（`disabled` 判定に演出中フラグを含める必要がある可能性）
- **フロントエンド 状態管理**: `frontend/src/stores/conversationStore.ts`（「今回新規到着したメッセージか、復元されたメッセージか」を判別する状態の追加。`isInputLocked` ゲッターの見直し）
- **フロントエンド 型定義**: `frontend/src/types/chat.ts`（演出制御用フラグを `Message` に持たせる場合、localStorage永続化に含めない設計が必要）
- バックエンド・DBスキーマへの影響なし。

## 機能要件

- FR-1: アシスタント応答テキストが新規に会話へ追加された際、`MessageBubble` はそのテキストを一括表示せず、先頭から1文字ずつ時間差で表示するタイプライター風アニメーションを再生する。
- FR-2: テキストのタイプライター表示が完了した後、該当メッセージに含まれるUIコンポーネント（`TableView`／`BarChartView`／`TrendChartView`／`StatCardsView`／`DonutChartView`／`ChoicesView`／`FaqListView`）はフェードイン（透明度のトランジション）で出現する。テキスト表示中はコンポーネント領域を表示しない。複数コンポーネントがある場合は全て同時にフェードインする（決定記録 D-3）。
- FR-3: 演出は「今回のセッション中に新規到着した」アシスタントメッセージにのみ適用する。サイドバーでの会話切替やページ再読み込みで localStorage から復元された既存メッセージは、テキスト・コンポーネントとも演出なしで即座に全体表示する（決定記録 D-1）。
- FR-4: 演出は `POST /api/chat` 経由の通常応答（`postChat`）だけでなく、フロント内で直接組み立てられるアシスタントメッセージ（`createAlertConversation` 等、`MessageBubble` を経由する全アシスタントメッセージ）にも同様に適用する。
- FR-5: テキストのタイプライター表示中からコンポーネントのフェードイン完了までの間、当該メッセージの選択肢ボタン（`ChoicesView`／`FaqListView`）はクリック不可のままとし、`MessageInput` も送信不可のままとする（現行の `isLoading` による無効化を演出完了まで延長する）。演出完了後は現行どおり有効化する（決定記録 D-2）。
- FR-6: 演出対象は `message.content` のテキストのみ。UIコンポーネント自体の内部データ（表のセル値、グラフの数値、選択肢の文言等）はタイプライターの対象外で、フェードイン時に完成形として一括表示する。
- FR-7: バックエンドAPI（`POST /api/chat`）のリクエスト・レスポンス形式、通信方式（1回のJSON応答）は変更しない。
- FR-8: 応答テキストが空文字の場合はタイプライター表示をスキップし、直ちにコンポーネントのフェードインへ進む。
- FR-9: `v-html` 等の生HTML描画は用いず、既存のXSS対策方針（`dynamic-ui-generation.md` 重大な非機能要件）を維持したまま文字送り表示を実装する。

## 受け入れ基準

- AC-1: メッセージ送信後、アシスタント応答テキストが一瞬で表示されず、1文字ずつ時間差で表示されるのが目視で確認できる。（FR-1。手動確認）
- AC-2: テキストの文字送り表示が完了した後にのみ、表・グラフ等のコンポーネントがフェードインで出現し、テキスト表示中はコンポーネントが見えない。（FR-2。手動確認）
- AC-3: 表とグラフの両方を含む応答（例:「今月の問い合わせ件数を担当者別にまとめて」）で、演出後に全コンポーネントが正しい内容で表示され、従来と表示内容そのものは変わらない。（FR-2, FR-6。手動確認）
- AC-4: サイドバーで別会話に切り替えてから元の会話に戻ると、過去のメッセージ（表・グラフ含む）はタイプライター・フェードインを再生せず、即座に全体表示される。（FR-3。手動確認）
- AC-5: ページを再読み込みすると、localStorage から復元された会話のメッセージは演出なしで即座に全体表示される。（FR-3。手動確認）
- AC-6: 演出中（テキスト表示中〜コンポーネントのフェードイン完了まで）は `MessageInput` が送信不可のままであり、当該メッセージの選択肢／FAQボタンもクリックできない。演出完了後は通常どおり操作できる。（FR-5。手動確認）
- AC-7: モニタリング画面のアラートから遷移して作成される会話（アラート文＋選択肢）でも、同様のタイプライター表示とコンポーネントのフェードイン演出が適用される。（FR-4。手動確認）
- AC-8: 通信エラー時・応答形式不正時のエラーメッセージ吹き出しも、他のアシスタント応答と同様にタイプライター表示され、画面はクラッシュしない。（FR-1, FR-9。手動確認）
- AC-9: `cd frontend && npx vue-tsc -b` と `npm run build` がいずれもエラーなく完了する。（自動検証: 型検査・ビルド）

## 制約

- 技術スタックは既存踏襲: フロントエンド Vue 3 + TypeScript + Vite + Pinia + Tailwind CSS 4。フロントエンド自動テスト（vitest等）は未導入のため、AC はビルド・型検査を除き手動確認が基準となる。
- バックエンドAPI・応答スキーマ（`docs/requirements/dynamic-ui-generation.md` で定義された `{reply, components}` 構造）は変更しない。実ストリーミング配信は導入しない（`dynamic-ui-generation.md` non-goals を踏襲）。
- 会話・メッセージは引き続き Pinia ストア＋ localStorage（`conversationPersistence.ts`）で永続化する。演出制御用に追加する状態（「新規到着か復元か」のフラグ等）は、永続化データ（`Message` の保存形）に含めて localStorage を汚さないよう設計すること（`isConversation` のゆるいバリデーションが通ってしまう点に注意）。
- 既存の `choicesEnabled`（`ChatWindow.vue`）・`isInputLocked`（`conversationStore.ts` ゲッター）のロジックは、演出完了状態を考慮するよう改修が必要になる（FR-5、決定記録 D-2）。
- Tailwind CSS のトランジションユーティリティ（`transition-*`）は既に `ChoicesView.vue` 等で使用されており、フェードインの実装もこの流儀に合わせることが望ましい。
- 過度な作り込みをせず「動くプロトタイプ」を優先する既存方針（`dynamic-ui-generation.md` 制約）を踏襲する。

## 決定記録

- D-1（旧 OQ-1）: 演出の適用範囲 → 新規到着メッセージにのみ再生し、会話切替・再読み込みでの再表示時は即時全体表示（推奨案どおり確定。FR-3, AC-4, AC-5）。理由: 既存の `choicesEnabled` ロジック（`message.id === latestMessageId`）が既に「最新メッセージ」を特別扱いする設計になっており自然に拡張できる。また会話を見返すたびに同じ演出で待たされるのはUXとして冗長。
- D-2（旧 OQ-2）: 演出中の入力欄・選択肢ボタンの無効化範囲 → タイプライター表示中からコンポーネントのフェードイン完了までの間、`MessageInput` と選択肢／FAQボタンを無効のまま維持（推奨案どおり確定。FR-5, AC-6）。理由: 一般的なチャットUIの慣習（応答が出揃うまで次の操作を抑制する）に沿う。応答受信時点で即時有効化する案だと、テキストがまだ表示途中の吹き出しの選択肢ボタンを押せてしまい、表示とインタラクションの整合が崩れる。
- D-3（旧 OQ-3）: 複数コンポーネントのフェードインタイミング → 全コンポーネント同時にフェードイン（推奨案どおり確定。スタガー表示は非スコープ。FR-2, AC-2, AC-3）。理由: 元のメモの「コンポーネントをふわっとフェードインさせる」は単一の演出として読め、最小実装で要望を満たせる。スタガーは見栄えの追加要望であり、実装・タイミング調整のコストが増える割に本要件の主眼（テキストのタイプライター＋その後のコンポーネント表示）に必須ではない。
